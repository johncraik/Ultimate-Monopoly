## 2. game-rules.md — new rule sections (dictated by John)

John dictated five further sections; each was tidied into the doc's house style
(intro line where useful, bold lead-ins, numbered lists, `---` dividers).

- **Loans** — last-resort borrowing. Loan amount = the shortfall rounded up to
  £50; keep up to £200 in account; max 3 outstanding loans; repay ≥5% of the
  *original* principal on passing GO/Free Parking (compounds to 10% if both
  passed); may overpay or pay early.
- **Mortgaging** — 10% of purchase cost repaid per mortgaged property on
  passing GO; applies to forced mortgages; only properties still mortgaged at
  the moment GO is passed are charged.
- **Reserved Properties** — early-game equaliser: nobody completes a set until
  everyone can. Landing on the final property of a set you otherwise own lets
  you *reserve* it at 50% cost; reserved property is inert (no rent, unbuyable);
  unreserve at full price once everyone has a set or a reservable one; the
  mechanic ends once every player holds a set.
- **Stations** — price scales with stations owned: £200 base + a surcharge
  equal to the rent for the combined count (2nd £250, 3rd £300, 4th £400); only
  currently-owned stations count; no surcharge for stations gained via Free
  Parking / card / deal.
- **Building Rules** — double hotels (cost of 5 houses, one per set, any number
  of sets); half-price building on a complete, never-built-on "street" (one
  side of the board).

**Corrections made during dictation:**
- *Loans rule 1* was first written as "even after mortgaging every property" —
  John corrected it: a loan does **not** require mortgaging first; a loan and
  mortgaging are alternatives the player chooses between. The "last resort"
  intro line was dropped.
- *Building Rules rule 3* — confirmed interpretation (b): the half-price
  discount is earned by owning a pristine (never-built-on) full street and then
  **persists** for all building across that street for as long as the whole
  street is owned; lost only if a set in the street is lost.

## 3. Full-rules review and gap resolution

Re-read the whole of `game-rules.md` and summarised the game back to John as a
comprehension check, then listed eight gaps the doc did not yet cover. John
resolved all eight, and the doc was updated accordingly:

- **Movement** — new section: players start travelling **clockwise**; a player
  always moves in the direction they are facing (own roll, third die, card
  effects alike); **any movement onto a space performs that space's action**,
  the sole exception being a swap with another player.
- **Double 6** — now also "performs that space's action", consistent with
  Doubles 3/4/5 and the new general movement rule.
- **Jail** — new section: leave by rolling a double, a card, or paying the fee;
  a double rolled to leave jail behaves as a normal double (move, then change
  direction); 3-turn limit (must leave on the 3rd); fee escalates £50 → £75 →
  £113 (previous + 50%, rounded up), resetting to £50 on the next jailing.
- **Bankruptcy** — new section: the game is won by the last player not
  bankrupt; bankruptcy may be declared voluntarily at any time (an "I quit"
  button); all money and properties return to the **bank**, never to another
  player; a creditor who triggered the bankruptcy is still paid the full
  rent/fine — by the bank.
- **Stations 1st-price** — confirmed: the first station is £200 base, no
  surcharge (already correct in the doc).
- **Utilities** — confirmed standard; no custom section needed.

## 5. Architecture discussion — building the helper app

An extended design discussion on how to build the actual game on top of the
existing config/social platform. Key decisions reached:

- **Helper, not simulator.** The app assists a *physical* board game. Players
  roll real dice and move real tokens; the app is the authoritative ledger for
  money, loans, cards, ownership, buildings, and all bespoke bookkeeping. It
  does track board positions (it must, to compute third-die movement).
- **Dice behind an interface** (`IDiceSource`) — manual input now, a virtual
  dice option as a later second implementation.
- **Rules engine as a separate, pure library** — no EF, no web, no `JC.*` web
  deps — so it is unit-testable. Command-in, events-and-state-out; deterministic.
- **Live game state ≠ persistence entities.** The engine owns a rich in-memory
  state object; the database stores serialised **snapshots** of it. The live
  state is *not* normalised into relational tables — it is a JSON document.
- **One snapshot per turn, kept (not overwritten).** A game becomes a timeline.
  This one mechanism covers persistence, crash recovery, the "game cache",
  **backtrack**, replay, and **analytics/graphs**. John extended the idea here:
  per-turn snapshots → a backtrack feature and stats pulled from the snapshot
  history.
- **Cards are a fixed, predefined set** — *not* user-authorable. This is the
  key reason the card system is now tractable: previous attempts failed because
  dynamic user-created cards forced a general-purpose effect interpreter. A
  closed set means one hand-written, individually tested handler per card.

A clarification on persistence: the database **does** hold the full game state
per turn (for save / load-later / crash survival) — the snapshot *is* that full
state. The decision is only about *shape*: one serialised document column
rather than a normalised table tree.

## 7. Game-engine data models (built by John)

John built the first game-engine persistence models:

- **`GameTurn`** (`Models/DataModels/Games/GameTurn.cs`) — `AuditModel`, GUID PK,
  FK to `Game`, composite FK `(GameId, UserId)` to `GamePlayer`, `TurnNumber`,
  `IsFinalTurn` (+ `SetFinalTurn`), `IsCurrentTurn` helpers. One `GameTurn` per
  turn; turns are snapshotted at the **start** of each turn (so a snapshot
  captures pre-turn state).
- **`GameSnapshot`** (`Models/DataModels/Games/GameSnapshot.cs`) — 1:1 with
  `GameTurn` (shared PK `TurnId`), FK to `Game`, `StateJson` holding the
  serialised game state.
- **`GamePlayer`** — modified: removed the live-state fields (`BoardIndex`,
  `Direction`, `Money`, `JailCost`, `TripleBonus`) and their movement/money
  methods, which now belong in the JSON snapshot per the engine doc. Added
  `OrderId` (turn order, 0–7, max 8 players) with `SetOrderId`. `Dice1`/`Dice2`
  (the permanent dice number) stay; `SetDiceNumber` now validates and returns
  `bool`.
- **`Game`** — added a `Players` collection navigation.

## 8. Model review and fixes

Reviewed the models. Findings:

- **Critical — invalid FK.** `GameTurn` referenced `GamePlayer` via a single
  `PlayerId`, but `GamePlayer` has a *composite* PK `(GameId, UserId)` — EF
  would fail the model build. **Fixed by John:** `GameTurn` now has `UserId`
  and a composite FK `(GameId, UserId)`.
- **Turn ordering split across two models** — `TurnNumber` was on `GameSnapshot`
  while `IsCurrentTurn` ordered by `CreatedUtc` (fragile DateTime equality).
  **Fixed by John:** `TurnNumber` moved to `GameTurn`; `IsCurrentTurn` now
  orders by it.
- **`GameSnapshot.StateJson`** had no `[Required]`. **Fixed by John** (added
  `[Required]`; explicit EF column type config deferred).
- **`.csproj` changed unexpectedly** — switched the `JC.*` libraries from local
  DLL references to NuGet `PackageReference`s and dropped explicit transitive
  deps. Flagged; John confirmed to **ignore** (intentional / not a concern).
- **DbSet registration** — `Game`, `GamePlayer`, `GameTurn`, `GameSnapshot` are
  still not registered in `AppDbContext` and have no migration. John is
  deferring this deliberately.
- Minor items noted (commented-out code left in `GamePlayer`, uninitialised
  `Game.Players` list, `IsCurrentTurn`'s `.Max()` throwing on an empty
  sequence) — left for John.
