## 4. GameSetupService — built out across the session

`Services/Games/GameSetupService.cs` grew from a stub to the full setup service.

- **`TryCreateNewGame`** — validates the board skin and the name (via
  `ValidateGame` + `ModelStateWrapper`), creates `Game` + host `GamePlayer` in a
  transaction. *Bug fixed:* QR generation was inside the transaction `try`,
  after the commit — a QR failure triggered a bogus rollback and reported
  failure on an already-created game. Moved the QR step outside the `try`.
- **`TrySetPlayerDiceNumbers`** — validates user + Setup state, sorts the dice
  pair, checks uniqueness, persists. A `lock` was attempted for the
  concurrency race and **removed** — it was sync-over-async, didn't actually
  close the race, and the lock object was per-instance on a scoped service.
- **`TryReorderPlayers`** — public (validates the posted set matches the live
  players) delegating to a private overload that renumbers `OrderId` 0..N-1.
- **`TryJoinGame`** — Setup-state + user validation, idempotent already-joined
  check, capacity check, block check, appends `OrderId = max+1`.
- **`TryLeaveGame` / `TryKickPlayerFromGame`** — consolidated the shared
  soft-delete + reorder logic into a private `TryRemovePlayerFromGame`. The host
  cannot leave; kick is host-only and targets a *different* user. (An earlier
  kick implementation conflated caller and target — fixed.) Player removal now
  **hard-deletes** the `GamePlayer` (no value keeping soft-deleted rows during
  setup).
- **`GetSetupGame`** — read method for the Setup page; includes `Players` +
  `BoardSkin`, and returns the game **only to the host**.

## 5. Dice-number uniqueness — Option A (DB unique index)

The custom rules give each player a permanent two-dice "number"; numbers must be
unique within a game. After weighing an app-level gate vs. a DB constraint,
John chose **Option A — a DB unique index**:

- `GamePlayer.Dice1` / `Dice2` made **nullable** (`ushort?`) — unset = `NULL`,
  and MySQL allows multiple `NULL`s in a unique index.
- `SetDiceNumber` **normalises** the pair to sorted order (min, max), so the
  index correctly treats `(3,5)` and `(5,3)` as the same number.
- Unique index `(GameId, Dice1, Dice2)` on `GamePlayer` (see §6).
- `TrySetPlayerDiceNumbers` keeps the in-memory pre-check as a fast path, with
  the unique constraint as the real backstop.

This supersedes the "accepted race" position from earlier in the session — the
race is now closed at the database level.

## 6. EF indexes across all data models

Added `[Index]` attributes (no fluent config) to every data model in the
assembly. EF already auto-indexes FK columns, so explicit indexes were added
only for the no-FK `UserId`/`CreatedById`/`ToUserId`/etc. columns, composites,
and the one unique constraint:

- `FriendRequest` — `CreatedById`, `ToUserId`.
- `BlockedUser` — `(CreatedById, BlockedUserId)`, `BlockedUserId`.
- `Friend` — `CreatedById`, `FriendUserId`.
- `BoardSkin` — `UserId`; `SharedBoardSkin` — `UserId`.
- `Game` — `(CreatedById, State)`.
- `GameTurn` — `(GameId, TurnNumber)` — **non-unique on purpose** (backtrack
  soft-deletes turns and reuses turn numbers).
- `GamePlayer` — **`(GameId, Dice1, Dice2)` UNIQUE**, plus `UserId`.

No unique indexes on the social join entities — they are soft-deletable and the
services `new` up fresh rows after a soft-delete, which a table-wide unique
index would collide with (MySQL has no partial index).

## 8. Join-by-code

John added a `JoinCode` to `Game` (prefix `"MP"` + a 5-char code from a
`JoinCodeGenerator`, total 7 chars), set via `SetJoinCode`, with a
collision-retry loop in `TryCreateNewGame`. New service method
`TryJonGameFromCode(joinCode, userId)` looks the game up by code and joins.

*Note:* the method name is misspelled — `TryJonGameFromCode` should be
`TryJoinGameFromCode`.

Discussed code generation: truncating a GUID is weak (hex-only, no uniqueness
guarantee); the chosen approach is a fixed unambiguous alphabet + crypto RNG
(`RandomNumberGenerator.GetItems`) with a DB collision-retry loop.

## 10. Game UI — the Razor pages

Decision: **server-render the pages now, layer SignalR on later** (the
`GameHub` does not exist yet). The project has **global authorization**, so no
`[Authorize]` attributes on new pages.

**`Pages/Game/Index.cshtml`** — Create New Game. Two columns: col-8 create form
(name, board picker with "Default board" = null skin, rounding-rule select),
col-4 rounding-rules explainer. Posts to `TryCreateNewGame`, redirects to the
Setup page. Navbar "New Game" link wired to it.

**`Pages/Game/Setup.cshtml`** — the setup screen. Header (name, board, rounding
rule); col player list + col join QR with the join code beneath it; Start /
Cancel footer buttons (placeholders). Player cards: profile circle + display
name, host border + "Host" badge, dice numbers or "Not Set", Set-Dice and
host-only Kick buttons. Interactions wired as ordinary POST handlers
(`OnPostKick` / `OnPostSetDice` / `OnPostReorder`) — with the host/self auth
checks the future hub would do. Added `GameSetupService.GetSetupGame` as the
read method.

**`Pages/Game/Join.cshtml`** — route `/join/{id?}`. With an id: joins that game,
shows a danger card on failure. With no id: renders a join-code entry box; the
input is trimmed + upper-cased and passed to `TryJonGameFromCode`. Both success
paths redirect to `/Game/Waiting` (page not built yet).
