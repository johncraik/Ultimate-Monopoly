## 1. Board state hues — `--bs-bg-opacity`

The dark-theme tints were too saturated. John spotted the `.bg-*` classes were
rendering at `--bs-bg-opacity: 1`. Fix: the Bootstrap contextual backgrounds
compute `rgba(var(--bs-*-rgb), var(--bs-bg-opacity))`, so we don't move *off*
the variable — we override its value. No `bg-opacity-45` utility exists
(Bootstrap ships 10/25/50/75/100), so set it once on `.space-body`:

```css
.space-body { --bs-bg-opacity: .45; }
```

Composites over the white `.board-space` behind it → the intended light tint.
Corrected the stale `-subtle` comment on `StateClass` too.

## 4. In-game prompt JS framework

Set up the multi-prompt client architecture (saved as
[[project_ingame_prompt_js_framework]]):

- **Coordinator** `wwwroot/scripts/InGame/game-play-hub.js` — owns the *single*
  `game-play` connection per page (`window.GamePlayHub`). Routes
  `PromptOpened`/`PromptClosed` (+ on-connect / `onreconnected` resync via
  `GetCurrentPrompt`) to registered handlers. Exposes `registerPrompt({type,
  onOpen, onClose})`, `on(event, cb)` for non-prompt consumers, and (added later
  this session) `invoke(method, …)` for hub commands.
- **Handlers** — one module per prompt type under `/Prompts/`; each
  `registerPrompt`s and self-filters by `prompt.playerId === ctx.userId`. No
  handler opens its own connection. `dice-roll-prompt.js` is the first.
- Load order: coordinator first; it defers `connection.start()` to
  `DOMContentLoaded` so handlers (and `play-state`) register first.

Generalised mid-session so the coordinator also runs on the host page
(`[data-play]`, no `userId`) — only `gameId` is required to connect.

### Live sync — re-fetch the rendered partial (not client repaint)

`StateChanged` ships the whole cache, but the SignalR JSON protocol serialises
enums as **numbers** and several display bits (`CanEndTurn`, jail flags) are
gate/`[JsonIgnore]` computations — a pure client repaint would duplicate
mapping + gate logic. So, matching the **setup-hub precedent**
(`OnGetPlayerCard` → `Partial`):

- Play body extracted into `_PlayView.cshtml` (model = `GameEngine` bundle).
- `PlayModel.OnGetStateAsync` returns `Partial("Play/_PlayView", Engine)`.
- `play-state.js` registers `GamePlayHub.on('StateChanged', …)` → re-`fetch`es
  `?handler=State` → swaps `.play-page` innerHTML (bursts coalesced). Board
  popovers re-init via `window.initBoardPopovers` (the board partial's own
  "re-init on live render" TODO, now done).

Trade-off recorded: a full swap resets accordion open-state / scroll each frame.
Acceptable for now; targeted sub-swaps are the future fix.

## 9. 🔴 CRITICAL BUG — `GameModel.GetPlayers` clockwise order is wrong

**Symptom (John, 3-player game).** Players A (seat 1), B (seat 2), C (seat 3).
Turn advances **A → C** (skipping B), then **C → A**, repeating. **B never gets
a turn.**

**Root cause — `GameModel.GetPlayers(povPlayerId, …)` does not sort by
`OrderId` on its main path.**

```csharp
var players = Players.Where(p => p.IsBankrupt == bankrupt).ToList();   // ← NOT ordered
...
var afterPov  = players.Where(p => p.OrderId > povPlayer.OrderId);     // filtered, not sorted
var beforePov = players.Where(p => p.OrderId < povPlayer.OrderId);     // filtered, not sorted
var list = new List<PlayerModel>(afterPov);
if (!excludePovPlayer) list.Insert(0, povPlayer);
list.AddRange(beforePov);
return list;
```

`afterPov`/`beforePov` preserve the **underlying `Players` list order**, which is
**not** guaranteed to be seat (`OrderId`) order — `GameSetupService.TryStartGame`
builds it via `game.Players.Where(p => !p.IsDeleted).ToList()` with **no
`OrderBy`**, so it's EF/insertion order (and a seat *reorder* during setup
changes `OrderId` without reordering the list). Only the *not-found fallback*
branch sorts (`players.OrderBy(p => p.OrderId)`); the normal path never does.

`TurnStateProvider.AdvancePlayer` calls `GetPlayers(excludePovPlayer: false)`
and walks the returned list to pick the next non-missing player. If the list
is, say, `[A, C, B]` (insertion order), then from A it picks **C**, not B — the
exact symptom. The list happens to look right whenever insertion order matches
seat order, which is why it "mostly works."

**Fix (next session).** Sort by `OrderId` on the main path — one line:

```csharp
var players = Players.Where(p => p.IsBankrupt == bankrupt)
                     .OrderBy(p => p.OrderId)
                     .ToList();
```

(Then `afterPov`/`beforePov` inherit the order; the fallback's `OrderBy` becomes
redundant but harmless.) Alternatively order `afterPov`/`beforePov` individually.

**Also worth doing:** add a `GameModel`/turn-advance unit test for the 3-player
A→B→C→A rotation (and a non-seat-order `Players` list) so this can't regress —
this is exactly the kind of ordering invariant `game-engine.md` §4
(determinism) wants covered. The sidebar player list also relies on this order;
seat labels use `OrderId` directly so they *looked* right, masking it.
