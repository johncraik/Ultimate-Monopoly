## 3. 🌟 Statistics design — `game-stats.md` rewritten

Long design discussion → substantial `game-stats.md` update:
- **Materialised read model.** The per-game-per-player summary is now
  **`PlayerGameStat`, materialised at conclusion** (not re-combed per view). The
  decisive reason is lifetime stats: aggregating `PlayerGameStat` rows is O(games),
  vs re-combing every finished game's blobs per profile view. Reconciled with the
  doc's old "on-demand" stance — §7.3 always blessed a wide per-game summary; the
  comb just runs once and persists. **Stat views read only `PlayerGameStat`**,
  never raw snapshots/events (so even graph *series* are persisted on it).
- **Projection job (§12).** `GameStatsService.ComputeForGameAsync` combs
  `GameTurnEvents` + `GameSnapshot`s → one `PlayerGameStat` per player, with a
  `StatsVersion` rebuild lever. Two Hangfire entry points: fire-and-forget from
  `ConcludeGame` (after commit) + a recurring `MissingGameStatsJob` safety-net.
- **Conclusion modes (§11).** A=Winner / B=Draw → stats projected; **C=Cancel →
  no stats**. Cancelled games are excluded for free (no row produced).
- **Profit = net** (accurate: `(rent+sell+mortgage) − (buy/auction+unreserve+build+unmortgage)`).
  **Fines** = `Tax + MortgageFee + FreeParkingPay + CardCharge` (later +`DiceNumBonus`).
- **Catalogue (§13)** expanded to ~60 metrics across 13 groups; John then curated
  it (`[X]` keep / `(…)` drop, dropped cards, added a few), and I tidied the doc
  to match.
- **Net-worth definition** pinned (see §8 below).

## 4. Architecture — engine owns the calc; `PlayerGameStat : PlayerStatRecord`

Decided the stats *computation* ships in **`MP.GameEngine`** (it interprets engine
receipts and values engine state via `ResolveSet` / `MoneyHelper` /
`PropertySetHelper` — all engine-side), keeping the engine pure and unit-testable
against the real JSON. Engine returns pure **`PlayerStatRecord`**; the web's
**`PlayerGameStat : PlayerStatRecord`** adds the EF keys/FKs/`StatsVersion`. Not
all-in-web (would duplicate engine semantics and drift).

## 7. 🌟 The twelve stats services — built and validated against the real game

Every service validated against the example JSON (per-player numbers reconciled,
and where possible cross-checked — e.g. spending buckets sum to `MoneySpent`,
income to `MoneyEarned − LoanTake`).

- **CashFlow** (John) — flagged the **missing `PlayerId` filter** (every player got
  the table total). Fixed → validated (`29a` earned 10400/spent 11750/net −1350;
  `653` 10250/8450/+1800).
- **Spending** (John) — buckets didn't reconcile to `MoneySpent`: the
  **`DiceNumBonus` outflow** ("your number rolled, you pay the roller") had no
  bucket. John folded it into fines → partitions exactly (11750 / 8450).
- **Income** (John) — **landing-on-GO money** was dropped; John included it in the
  GO money while keeping `TimesPassedGo` passing-only, and renamed the field
  `MoneyFromGo`. Reconciles to `MoneyEarned − LoanTake` (loan disbursement
  deliberately excluded; lives under loans).
- **Property** (the hard one) — John's first pass had per-turn (not cumulative)
  profit, rent-paid pollution, a line copy-paste, and a set over-count. I
  **refactored it clean** (declarative signed-sum accumulation, touched-only pool,
  argmax once). Then researched "complete set" → `PropertySetHelper.GetOwnedSets`,
  and per John's call: set profit restricted to **ever-completed** sets,
  **`onlyBuildable: false`** (stations/utilities count). Validated (`29a` most prop
  21/+1000, least 24/−1250, most set Red/+300, least Yellow/−1300).
- **Movement** (John) — flagged a critical `||`→`&&` (the whole movement loop was
  dead), a missing `PlayerId` filter, order-sensitive dice-number matching, and the
  distance logic. John fixed the first three; on distance I'd initially proposed
  the wrong model — John corrected: **clockwise/anti = the player's facing
  (`PlayerDirection`)**, signed by travel direction (counter-direction
  *subtracts*, so forward-3/back-3 nets 0), magnitude = shorter arc. Applied →
  validated (`29a` cw 151/anti 32; `653` 76/77).
- **Jail** (me) — `PlayerEnteredJailReceipt` is now wired (John). `TimesSentToJail`
  = that receipt; `ByPaying` = `JailFee` count; `ByDice` = total leaves
  (`PlayerMoved` *out of* jail) − paying − card; `TotalJailTurns` = own turn-starts
  on the jail space. Validated.
- **FreeParking** (me) — `PropertyTransferReceipt` reason `FreeParking` (±Value) for
  handed-in/taken; `FPHandedInSets` (append-only) → JSON. Caught **my own
  enum-order assumption** here (`FreeParking = 4`, not 3). Validated.
- **Loan** (me) — `LoanTake`/`LoanRepay`/`Mortgage`/`Unmortgage`/`MortgageFee`
  receipts + final-snapshot loan list for outstanding/repaid. (`653` took the
  loans, not the bankrupt — another don't-assume.) Validated.
- **Card** (John) — no-op `return record`; the card fields are deferred (§14.2).
- **Endgame** (me) — bankruptcy fields from `PlayerBankruptedReceipt`, `TurnsSurvived`
  = bankruptcy turn or final turn, `FinalBalance`/`FinalNetWorth`. John simplified
  to use the passed `player` directly (it's already the final-turn player) and made
  `Bankrupted = player.IsBankrupt || receipt`. Validated.
- **StateOverTime** (me) — `PeakNetWorth`/`PeakBalance` (+ earliest peak turn) over
  every snapshot. Validated (`29a` peak 6685 @40 then crashed to bankruptcy @43).
- **Graph** (me) — the four per-turn series, one value per snapshot, **breaking at
  the first snapshot showing the player bankrupt** (start-of-turn snapshots, so it
  captures up to the bankruptcy turn). `WealthRank` = 1 + players richer that turn.
  Validated (`29a` 37 points to turn 43; `653` 38 to turn 44).

## 8. 🌟 Net-worth — shared static helper

Net-worth definition (John): **balance + mortgage value of NON-mortgaged
properties + sell value of all buildings** (a mortgaged property's value is
already realised as cash). Built **`StatisticsOrchestrator.CalculateNetWorth(player,
game, board, roundingRule)`** (static, shared by Endgame / StateOverTime / Graph),
with building sell value = build-steps-above-SET × one building's sell value
(double hotel = 10 steps). Added **`CompleteGameSnapshot.RoundingRule`** (needed by
`MortgageValue`). Flagged choices: reserved counts as non-mortgaged; building sell
value uses the base (non-street-effect) rate.
