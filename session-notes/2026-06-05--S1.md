## 4. 🌟 `PropertyTransferService` — the property-move mirror

The session's first architecture build. John wanted a centralised place to move
properties between player / bank / Free Parking (for deals, cards, FP), analogous
to `TransactionService` for money.

- **Naming** (design Q&A): the existing `PropertyTransactionReceipt` had already
  half-decided the vocabulary, and `TransactionCounterparty` (Bank / FreeParking /
  Player) is a shared, generic endpoint enum. Landed on **`PropertyTransferService`**
  + **`PropertyTransferReason`** (Buy, Reserved, Auction, Deal, FreeParking,
  ReturnToBank, Bankrupt). Renamed the receipt **`PropertyTransactionReceipt` →
  `PropertyTransferReceipt`**, added a `Reason` field (mirroring
  `FinancialTransactionReceipt`), updated the `[JsonDerivedType]` discriminator
  (`"PropertyTransfer"`) and the docs (`transactions.md`, `event-receipts.md`).
  John aligned the player field to `CounterpartyPlayerId`.
- **The service** (I built it): one method per reason (`Buy` / `Reserve` /
  `WinAtAuction` / `TakeFromFreeParking` / `HandIntoFreeParking` / `ReturnToBank` /
  `Transfer` / `Bankrupt`), a private `Move` core, `ApplyOwnership` (≈
  `ApplyBalances`), and an identical two-perspective `EmitReceipts`. **Title only**
  — it flips `PropertyModel` state + emits the receipt, and deliberately leaves
  `NormaliseRentLevels` and the reserve-rule break-through to the caller, exactly
  as `TransactionService` leaves downstream effects to callers. Synchronous (`void`)
  — no prompts/shortfall in a title move. Registered scoped.
- **Refactor:** John pointed `PropertyService`'s three title-move sites (`Buy`,
  `WinAtAuction`, `Reserve`) at it (old code left commented). Review: correct and
  faithful, reference-safe (no mid-turn `SaveChanges`, even across the auction
  prompt await), and **no DI cycle** (`PropertyService`'s closure is
  Auction/Transaction/PropertyTransfer — none reach `FreeParkingService`).
  Mortgage / unmortgage / un-reserve correctly stay direct — they're same-owner
  state flips, not transfers, so they emit no transfer receipt (John confirmed).

## 6. 🌟 Rule Citations — designed and built end to end

The session's big new feature: explain "wtf just happened" by surfacing the rules
that fired this turn.

- **Design discussion → `design-docs/rule-citation.md`** (new, 15 sections). Key
  decisions: a **sibling stream to event receipts** (not a field on them — rules
  and receipts aren't 1:1); the **engine emits codes, the web resolves text**
  (engine stays pure); **stable semantic keys, not positional numbers** (the
  keys-not-indexes lesson from `choice-events.md` §15.9); **granularity follows the
  engine's branches** (a node is citable when the engine distinctly applies it);
  **static text only, no templating** (helper-not-simulator — cite the rule, the
  table fills in who/why); citations **ship in the live `StateChanged` frame** (the
  inverse of receipts, which are `[JsonIgnore]`d); drift guarded by a **hand-written
  enum + lockstep test** (source-gen evaluated and dropped at this scale).
- **`RuleCode` enum** (I wrote it): 57 codes — the citable subset only — ordered by
  `game-rules.md` section, each commented with the rule + engine site. Non-citable
  rules and the card/deal subsystems are excluded.
- **The seam divergence (John):** dropped a full `IRuleEmitter` (overkill for
  "append an enum") — `GameEngine.CiteRule(code)` (internal) → `cache.AddRuleCode`,
  with `RuleCodes` (not `[JsonIgnore]`d) and `ClearRuleCodes` at the turn boundary
  (alongside `ClearEvents`).
- **John wired `CiteRule` across every service.** Deep-dive review: coverage
  complete (all 53 non-bankruptcy codes cited; the 4 `Bankruptcy_*` await the
  service). Three placement findings — two **withdrawn** as intended (the FP
  pay-fee + no-pay-on-double pair gives full context on a double; the triple
  movement/no-direction-change cites apply on a 3-triples-to-jail too, since a
  triple never flips direction or grants third-die movement regardless). The third
  was real and **fixed:** `NormaliseRentLevels` cited `Mortgage_NoSetRentWhileMortgaged`
  / `Reserved_PropertyInert` / `Utility_PairMultiplier` on *standing state* every
  recompute — moved to the action/observation points (mortgage → `MortgageProperty`;
  utility → `PropertyRent`; reserved already in `PropertyRent`). Also spotted the
  **triple bonus is now wired** (`PlayerService.ResolveTripleBonus` + `Triple_Bonus`
  cite).
- **`RuleCatalog`** (web, John) — loads `rules.json`, resolves codes → `GameRule`.
  Review: the panel branch must preserve **cited order** not rulebook order; cache
  the parsed list; make `Point` a `char?`; register **singleton** (else the
  instance cache is a no-op). John fixed all (and the `RawRuleCode`/`<NONE>` →
  nullable-enum handling is clean).
- **`rules.json`** (I wrote it) — 57 entries, one per `RuleCode`, section-ordered,
  PascalCase, `RawRuleCode` matching each enum name; player-facing standalone text
  meaning-matched to `game-rules.md` and the engine. (The full rulebook — the
  `"<NONE>"` rows for the rules *page* — is later work.)

## 7. The "Rules Occurred this Turn" panel + sidebar layout

- Replaced the host page's **"Current Player"** sidebar pane with a **Rules** card
  in `_PlayView`: injects `RuleCatalog`, resolves `cache.RuleCodes` (reversed →

  `StateChanged` and clears each turn boundary.

  **`flex-basis: 0`** (players `flex: 2 1 0`, rules `flex: 3 1 0`) so the panes size
  to their share of the column height — the board stays tallest, the sidebar
  stretches to it, and the list scrolls inside its pane. Same change gave the
  40/60 ratio.
