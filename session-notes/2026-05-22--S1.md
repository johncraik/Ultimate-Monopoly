## 2. SignalR — two hubs, two groups

Agreed: **`GameSetupHub`** (group `game-setup__{gameId}`) and **`GamePlayHub`**
(group `game-play__{gameId}`), splitting on the setup → in-play phase boundary.

Key point clarified during discussion: **SignalR groups are namespaced per
hub** — a group named the same in two hubs cannot collide. So the distinct
group names are a *clarity / logging* choice, not a correctness requirement;
isolation is already free from the hub split. The `StartGame` handoff (setup
clients reconnecting to the play hub) is the real design risk, deferred with
`GamePlayHub`.

---

## 3. Hub shells — `GameBaseHub` / `GameSetupHub` / `GamePlayHub`

Extracted a shared abstract **`GameBaseHub : Hub`** holding the connect/
disconnect plumbing; each derived hub supplies a `GroupPrefix`, and the base
builds `{prefix}__{gameId}`. `GameSetupHub` / `GamePlayHub` each also expose a
`public static GroupName(gameId)` for broadcasters.

- `OnConnectedAsync` reads `gameId` from the connection query string and, before
  joining the group, **verifies the caller is a `GamePlayer`** in that game via
  `GameService.CheckUserInGame` — aborts the connection otherwise. This closes
  the eavesdropping gap: group membership is the only thing gating server→client
  broadcasts.
- **Auth:** the project's global auth is a `FallbackPolicy`, which applies to
  *every* endpoint without its own auth metadata — including `MapHub<>`
  endpoints. So `[Authorize]` on a hub is redundant; it was dropped.
- **`IUserInfo` is not populated inside a hub** — `UserInfoMiddleware` is HTTP
  middleware and a hub invocation runs in its own DI scope. The hub uses
  `Context.UserIdentifier` instead (same as `PresenceHub`). `GameBaseHub`
  briefly injected `IUserInfo`, then dropped it.

`GamePlayHub` is a bare shell — no in-play pages or mechanics exist yet.

## 9. Snapshot model design — decisions reached

- **One `GameModel`** serves as both the live in-memory model and the
  serialised snapshot — *not* two models. The doc's "different things" is
  engine state vs. EF entities, not live-vs-snapshot.
- **The board is fixed and cached per game, never written into a turn
  snapshot.** `PropertyModel` carries only per-game mutable state (owner,
  `PropertyState`, `RentLevel`, `HasBeenBuiltOn`); price/colour/rents come from
  the cached `Board`. A deleted board skin falls back to the default board.
- **No ruleset versioning** — deliberate; a rule change is a code change that
  affects active games anyway.
- Jail has its own out-of-band board index (`100`), distinct from Just Visiting
  (`10`) — so `IsInJail` is unambiguous from position.
- Per-player state completed: direction, doubles/triples-in-a-row counters,
  triple-bonus accumulator, jail counter + cost, missed-turn count,
  passed-GO flag, per-player Free-Parking hand-in tracking.
- `PropertyColour` → renamed `PropertySet` (colour groups + Station + Utility);
  `RentLevel` reworked; `PropertyState` gained `Owned` and `FreeParking`.
- Free Parking pot is a single `uint FreeParkingAmount` on `GameModel`;
  properties "in" Free Parking are just `Properties` entries with
  `State = FreeParking`.

## 11. `GameCacheModel` + concurrency

The object held in the game cache is **`GameCacheModel`**, not a raw
`GameModel`. It wraps the `GameModel`, the `Board`, and the mid-turn `Events`
log, plus a `ConcurrencyStamp`.

- `GameCacheModel` follows a `DbContext`-style design. The `Game` property hands
  out a **tracked working copy** — `_working ??= new GameModel(_game)`, a deep
  copy made on first access and the same instance on repeat access — and
  `SaveChanges()` promotes that working copy to the committed `_game`, clears the
  tracker, and re-stamps. Forgetting `SaveChanges()` discards the work, by design.
- *Bugs caught during the cache rework:* an interim `SaveChanges()` re-cloned the
  original and saved nothing; the working-copy getter copied the
  primary-constructor parameter `game` instead of the committed `_game` field
  (reverting to turn-1 state after the first save); and in the deep-copy
  constructors `GameModel.Metadata` and `PlayerModel.FPHandedInSets` were shared
  references, defeating isolation. All fixed.
- **Concurrency stamp:** a stamp lives on the cache; a command carries the
  stamp it last saw; a mismatch rejects the command as stale; the new stamp is
  broadcast on change. An earlier "stamp in an encrypted cookie" idea was
  rejected — the stamp is game state, not per-device, and isn't a secret.
- Copy constructors were chosen over a `Deserialize(Serialize())` round-trip —
  more readable, at the cost of field-by-field upkeep (cards in lockstep).

**Loading / caching model (discussed):** the cache is a single shared
server-side instance per game; load is cache-first, DB snapshot on miss; a
per-game lock must serialise command application; eviction policy must be
deliberate (an evicted mid-turn game loses the turn — accepted).

## 13. Card system — design discussion (not coded)

Cards will be composed from a fixed, closed vocabulary: ~8-10 **Actions** (what
happens) and ~6-8 **Conditions** (when). "Branching" is **options** — a
pick-1-of-N player choice over Actions (e.g. "go to jail OR pay £500"), which is
declarative, not control flow. Discussed: card *definitions* are fixed and
belong in a catalogue (like the board), not the snapshot; an option-card pauses
for a player decision — the same interaction shape as buy/decline and NOPE, so
it should reuse one shared decision mechanism. `card-decks.md` to be written
before cards are coded.

## 15. Game cache service (web app)

`UltimateMonopoly` gained **`GameCacheService`** — wraps `IMemoryCache`, keyed
per game, 12h expiration, loads via `GameSnapshotService` on a miss.

- *Bug caught & fixed:* an early version held a single `_game` instance field,
  so `GetGame(gameId)` returned whatever game loaded first regardless of the id
  (and as a DI service the field was either useless per-request or one-game
  app-wide). Removed — the service is now stateless; per-game state lives
  entirely in `IMemoryCache`.
- `SaveChanges(gameId)` fetches the cached `GameCacheModel` and calls its
  `SaveChanges()`. `IMemoryCache` holds the reference, so mutating the cached
  object *is* updating the cache — no re-`Set` needed.
- Flagged, not yet addressed: `GetOrCreateAsync` isn't atomic (a per-game lock
  is wanted for the "one shared instance" guarantee); 12h *absolute* expiry can
  evict an active game mid-turn (prefer explicit removal / sliding expiry).

## 16. Engine architecture — pure logic, web does the wiring

Clarified and settled:

- `MP.GameEngine` is **pure logic** — functions are handed the game state
  (`GameModel` / `GameCacheModel` / a command) as parameters and return new
  state + receipts. The **web app does all the wiring**: load from cache → call
  the engine → save.
- An `ISnapshotService` abstraction was created, then **removed** — the engine
  never loads or saves (it's handed the state), so it needs no snapshot
  abstraction. Persistence lives entirely in the web app, where
  `GameCacheService` + `GameSnapshotService` already sit.
- **No DI container inside the engine.** Engine services have no external /
  swappable / lifetime-bound dependencies, so a container has nothing to manage.
  They're plain instance classes using constructor injection — `new`'d directly,
  not container-resolved. Keeps the engine self-contained and trivially testable
  (`new` the service, call it, assert — no `IServiceProvider`, no mocks).
