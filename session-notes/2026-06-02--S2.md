## 3. The three directives (now standing conventions)

John's answers hardened into three rules, saved as memories:

1. **The partial loads and live-updates from `GameCacheModel`** — same render
   path as the host page (server-rendered partial + state re-fetch swap).
2. **Anything you can't reach a value/command for: mock it with a `// TODO`,
   leave the button an always-present no-op — but the mock/stub still wears its
   real guard** (hide where state forbids, disable where the gate says no).
   Saved as `feedback_mock_ui_keeps_real_guards`.
3. **Every *computed* money figure shown goes through `MoneyHelper`** so it
   matches the game's rounding rule — `NormaliseAmountToPositive` for cost
   contexts (it `Math.Abs`es). Exact ledger values (Money, loan Amount/PaidBack)
   shown raw. Saved as `feedback_frontend_money_via_moneyhelper`.

Plus: use `PlayerCacheService.GetPlayerProfile` for the whole
`UserProfileViewModel` (the established pattern).

## 4. The player profile partial — built

Mirrors the host page's render architecture exactly:

- New `PlayerProfilePlayViewModel(GameEngine Engine, string PlayerId, string
  ViewerUserId)`. `ViewerUserId` (self or host) gates the command buttons via the
  host-bypass-aware `Can…` checks — never chooses what data is shown.
- `Player/Index.cshtml.cs` now resolves the engine via `IGameEngineFactory`,
  exposes the VM, and adds `OnGetStateAsync` (handler=State) returning the
  partial. A shared `LoadAsync` does auth (`CheckUserInGame` + the EF player
  lookup) + cache load (InPlay check, NotFound on a hydrate miss). Previously the
  page touched only EF; it now renders from the cache like `PlayModel`.
- `_PlayerProfileView.cshtml` (the partial) + `player-state.js` (re-fetch on
  `StateChanged`, **preserving the active tab** across the innerHTML swap) +
  `player-page.js` (delegated Roll/End-turn commands; `data-noop` stubs left
  unbound) + `player-profile.css` (the bottom glassy sticky icon tab bar and the
  "space rectangle" owned-property cards — colours via theme CSS vars only).
- The header Roll/End-turn button reuses the host's three-state machine but only
  renders when this profile **is** the current player; otherwise a whose-turn
  indicator. Disable-on-click; the next `StateChanged` frame re-renders it.

**Wired real:** money, jail cost, triple bonus, FP handed-in sets
(`player.FPHandedInSets`), owned-by-set grouping + SET badge, un-reserve = full
purchase price, mortgage-fee-at-GO (10%), loan table + 5% instalment, per-set
build cost / double-hotel (from the real `BoardSpace.BuildCost`).

**Mock (`// TODO`):** %cap (John to supply the func), houses/hotels-left and
total-build-cost (no building inventory in the model), mortgage/un-mortgage
*amount* (no engine formula). **No-op stubs** (gate-guarded but no command):
leave-jail, declare-bankruptcy, build/sell, mortgage/unmortgage, unreserve,
take/pay loan, deal. Cards tab is a blank "you have no cards"; Deal a "coming
soon" stub.

One flagged judgement call: the loan instalment uses `Math.Ceiling` then
`MoneyHelper`, but the rule says round **up** while `NormaliseAmount` rounds
to-nearest — so it's an approximate display figure (TODO-noted). A dedicated

## 5. Host drawer reuse + prompt auto-open/close — built

John's next ask: the drawer should **auto-open on the prompted player showing
the prompt, and auto-close when the prompt resolves**. Implemented:

- **New coordinator channel** `GamePlayHub.observePrompts({onOpen,onClose})`
  (`game-play-hub.js`) — type-agnostic, fired from the same
  `dispatchOpen`/`dispatchClose` path as the typed handlers, so it also covers
  the **reconnect/initial resync** (host loading mid-prompt opens the drawer
  correctly).
- **`player-drawer.js`** (new, host-only) owns the drawer: loads the real
  `/player-profile` partial (`?handler=State`, host as viewer → host-bypass gates
  light up), opens manually (the "View" buttons, with prev/next browsing) **or**
  prompt-driven (`onOpen` → open on `prompt.playerId`; `onClose` → close), and
  refreshes the open drawer on each `StateChanged` (tab preserved). Manual
  close/step is suppressed while a prompt is driving it. Drawer logic moved out
  of `play-page.js` (now just turn commands).
- **Prompt handlers self-filter** `if (hubCtx.userId && prompt.playerId !==
  hubCtx.userId)` — the tablet (no `userId`, the controller) renders **every**
  prompt and answers via host-bypass; phones stay scoped to their own. The three
  prompt partials + handler scripts are now included on the host `Play.cshtml`.
- **Layering** (`play.css`): drawer z-index dropped **1085 → 1052**, the band
  between Bootstrap's modal-backdrop (1050) and modal (1055). The page dims, the
  drawer stays **bright** (showing whose prompt it is), and the prompt modal
  layers on top. A prompt-driven open skips the drawer's own backdrop (the modal
  supplies it); a manual View open keeps it.
- (John also linked `player-profile.css` into `Play.cshtml`'s styles — the drawer
  now renders the same bottom-nav / space-rectangle partial, so it needs that CSS
  too.)

Interpretation flagged to John: the prompt shows as the existing **centered modal
over the open drawer** (reusing the phone prompt partials as-is), not re-rendered
*inside* the drawer body — the drawer supplies the "which player" context behind
it. He was happy with that on test.
