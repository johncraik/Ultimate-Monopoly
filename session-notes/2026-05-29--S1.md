## 2. IndexHelper tests → MoveIndex / AdvanceIndex

Wrote `HelperTests/IndexHelper_Tests.cs` — lots of scenario coverage for the
board-movement maths. This grew to 73 cases and surfaced two things:

- **A GO-pass-counting bug** (forward landing-on-GO counted as a pass; forward
  departing-GO didn't) — see §3 for the contract saga.
- **An infinite loop on negative `spaces`.** John spotted it from
  `index:6, spaces:-12 → throw` — the old `int spaces` while-loop kept ±40-ing an
  out-of-range value forever.

John's fixes (both better than what I'd have done):
- **`MoveIndex` is now a bounded step-simulation.** It walks one space at a time
  for exactly `|spaces|` iterations, counting a pass when it steps onto space 1
  (forward) or wraps 0→39 (backward), and treating arrival on 0 as a landing
  (no pass).
- **Negative spaces handled in the helper, not the caller.** `MoveIndex` keeps a
  signed `int spaces`; if negative it normalises to positive and **flips the
  direction** before the (bounded) loop — so a back-12 moves back 12, with no
  infinite-loop risk. `MovePlayer` dropped its `Math.Abs` and just forwards
  `amount`. One source of truth for signed movement.
- **`AdvanceIndex`** short-circuits `desiredIndex == GoSpace` to `(0, false)`
  (landing on GO is never a pass), and adds `currentIndex == 0` (departed GO) to
  the pass check.

Added negative-movement tests: concrete back-movement scenarios (Double 6
back-12 over/onto GO, Double 3 back-leg) plus an equivalence theory —
`MoveIndex(x, -n, dir) == MoveIndex(x, n, opposite(dir))`.

## 3. The GO-pass contract (I got it wrong twice)

The maths only made sense once John stated the rule plainly:

> **Passing GO** = your move's journey involves GO without *ending* on it.
> Crossing over GO (`35 → 42`) is a pass; **departing** GO ("I'm on GO, I
> rolled, I've passed GO", `0 → 5`) is a pass; **landing exactly on GO**
> (`35 → 40`) is **not** — that's a landing (and lands take a GO card, deferred).
> Direction-symmetric.

I first assumed landing-on-GO *was* a pass, then over-corrected to "departing
isn't a pass". Both wrong. Lesson restated: **confirm the rule with the
rules-author before asserting test expectations** — the tests are only as right
as the contract behind them. (Consistent with the existing
[[project_dice_turn_rules]] / "John's interpretation stands".)

## 8. GameExecutor — dirty cache + fatal-fault force-quit

Closed the `GameExecutor` dirty-cache TODO. On a work-item throw the pump now:
completes its channel (drops queued work — don't keep running a haywire game),
calls back to `OnPumpFaulted` which **removes the pump and
`GameCacheService.Invalidate(gameId)`s the cache**, then breaks. The next
`Enqueue` (now a small retry loop) spins up a fresh pump whose first item
re-hydrates from the last snapshot.

Added a **`GameFaulted` SignalR broadcast** to the same fault path: the executor
fire-and-forgets it to the `game-play__{gameId}` group, and `game-play-hub.js`
shows a centred, static-backdrop danger modal and redirects to `/Index` —
force-quitting every connected client (tablet + phones). (Also fixed a stray `e`
typo that would have thrown on load and broken the whole coordinator.)
