## 1. The DI / cache-access question

Opened on "how does a rule service get at the cache for a given gameId?".
The starting shape was per-method factory call inside `DiceService`. The
options John laid out:

1. **Factory injected; every method takes `gameId`**, opens with
   `_engineFactory.GetAsync(gameId)`.
2. **Scoped DI accessor** the hub populates before calling rule services.
3. **One bundle per game**, with rule services pre-built around the cache
   like the framework providers are.

Settled on option 1, with one refinement: rule services take `gameId` at
the *public* boundary; private helpers and inter-service calls take the
`GameEngine` bundle directly. The bundle is the per-call session
context — fetching once at the seam, threading through inside, never
re-fetching. The factory boilerplate (`GetCache` private wrapper) drops
out.

This is consistent with the engine architecture in `game-engine.md` §3
(interface-in-engine / impl-in-web), and with the existing
`GameEngineFactory` shape — the factory is the seam, nothing more.

## 2. Invocation model — the deadlock that wasn't obvious

Surfaced a real issue under the surface: if a hub method `RollDice(gameId)`
awaits `_diceService.RollTurnDice(...)`, and that awaits a `DiceRollPrompt`,
the hub invocation pipeline is blocked. SignalR's default
`MaximumParallelInvocationsPerClient = 1` means the same client's
`SubmitPrompt` invocation queues behind the in-flight `RollDice` — and
since the prompt can only resolve once the client submits, it's a
deadlock by construction.

Three resolutions discussed:

1. **Bump `MaximumParallelInvocationsPerClient`** — papers it over, leaves
   the bigger issues (long-lived hub invocations, no per-game
   concurrency coordination, disconnect mid-prompt).
2. **Per-game `SemaphoreSlim` + raised parallelism** — minimum-viable.
3. **Per-game executor** (channel-pumping actor) — the right cut.

Option 3 collapses three open items at once:
- The "per-game lock" item from earlier sessions (`game-engine.md` §14 /
  carried-over) — single-writer by construction.
- The "long await on the hub" problem — rule services run on a worker
  thread, not the connection thread.
- The cache's working-copy semantics — single-writer is now a structural
  invariant, not a convention.

`SubmitPrompt` is the exception: must stay out-of-band, calls
`IPromptProvider.TrySubmit` directly. That gives the cache a narrow
thread-safety obligation around `PendingPrompt` + `ConcurrencyStamp`
clearing; everything else stays single-writer on the executor.

Not implemented this session — but the future shape is clear: rule
services take `GameEngine` (not `gameId`), the executor resolves the
engine once on dequeue, and hubs become thin "enqueue and return". The
factory becomes an implementation detail of the executor.

## 3. The coupling concern and the three-layer cut

John spotted that the rule services were getting heavily coupled
(`DiceService → DoubleDiceService → MovementService → BoardService`) and
asked if response objects + a `PlayerTurnService` orchestrator would be
cleaner. My first framing — "pure resolution objects for dice, imperative
for cards" — got pushed back on correctly: dice doubles/triples also
draw cards (`game-rules.md` Doubles general rule 1, Triples rule 1), and
the cards can have immediate effects → not pure data.

Reframed to the actual right cut:

- **Pure data** — `DoubleEffects` is a static lookup table keyed on
  dice value. No service deps. One entry per double, naming the title,
  body, movement steps for roller and others, and the
  miss-turn / extra-effect flags.
- **Primitive services** — Movement, Balance, Card, Prompt-helpers. Each
  takes the engine; each does one narrow job.
- **Orchestrator** — `PlayerTurnOrchestrator` reads the data, calls the
  primitives in rule order, opens prompts itself.

`DoubleDiceService` collapsed into `DoubleEffects`. The per-double methods
(`DoubleOne` … `DoubleSix`) became one loop in the orchestrator.
`DiceService` reduced to "prompt → cache → receipt" — a primitive, not a
coordinator.

## 5. Engine error policy — read vs mutation

Started the session with one memory:
[[feedback_error_handling]] — "prefer bool/result returns for expected
failures". John clarified mid-session that this is the *web* convention;
the engine has the opposite contract. Saved
[[feedback_engine_error_bubbling]] — throws bubble all the way up, the
web layer rolls back the turn from the snapshot.

Then refined again a few exchanges later. The original engine memory
said "throws and bubbles, no false/null returns". John sharpened it:

- **Reads / extractions** (lookups, getters) → throw on "not found", bubble.
  The web rolls back via snapshot reload.
- **Mutations against the rules** → silent no-op. Don't throw, don't
  return false — just refuse. Throwing on illegal mutations forces the
  web layer to roll back perfectly-recoverable scenarios.

Updated the engine-error memory to capture both modes. Two of my
mid-session flags retracted as a result:
- `PropertyModel.MortgageProperty()` silent-returning on unowned property
  is correct (mutation against rules).
- `GameModel.GetPlayer` mixing throw + null is correct *if* we treat
  "doesn't exist at all" as a read failure (throw) and "exists but
  doesn't match the bankrupt filter" as a domain-meaningful null (not a
  failure).

The split lines up with how `GameCacheModel`'s working-copy semantics
already work — successful mutations land on `_working`; the engine never
needs to roll back per-mutation because the snapshot is the recovery
boundary.

## 6. `GameModel` primitive consolidation

John spent a long stretch consolidating helpers on the snapshot models.
Now landed:

- `CurrentPlayer(bool bankrupt = false)` — delegates to `GetPlayer`.
- `GetPlayer(playerId, bool bankrupt = false)` — throws on
  not-in-list-at-all, returns null on bankrupt-state-mismatch.
- `GetPlayers(bool bankrupt = false, bool excludePovPlayer = true)` and
  the POV-explicit overload — returns players ordered clockwise from a
  POV.
- `CheckAnyDiceNumbers(DiceRoll)` — returns the playerId whose dice
  number matches the roll (Dice Rolls rules 2 & 3).
- `GetPropertySpace(boardIndex)` — single-property lookup.
- `GetOwnedProperties(set?, includeMortgaged?, includeReserved?)` —
  the set filter is the new addition.
- `HasStreetEffect(set, absoluteCheck = true)` and the POV-explicit
  overload — collapsed via `PropertySetHelper.StreetPartner` (the
  dictionary suggested last review) and the new `StreetRuleQualifier`
  enum.

The big shift was `HasStreetEffect`:

- I'd misread the `absoluteCheck` default direction. John clarified: the
  default (`true`) is the strict "currently-qualifying-for-the-discount"
  check (no mortgaged / no reserved); `false` is the lenient "you'd
  qualify if X was unmortgaged" check used for UX hints. The same method
  serves both contexts.
- John then replaced the `bool HasBeenBuiltOn` on `PropertyModel` with a
  `StreetRuleQualifier` enum (`None`, `NeverBuiltOn`, `BuiltOn`,
  `Qualified`). `HasStreetEffect`'s body collapsed to a single
  `properties.Count(...) == streetIndexes.Count` line.

The `Qualified` state is sticky once earned (per Building Rules rule 3:
"keeps the discount for as long as they continue to own the whole
street"); transitions are the property-service's job when ownership
events fire. State machine is per-street but the enum is per-property —
the partner-property bumps happen as a unit.

## 7. `PropertyModel` state methods

John added the lifecycle mutators directly on the model:

- `ReturnToBank()` / `HandInToFreeParking()` — clear ownership, reset
  qualifier to `None`, reset rent level. Both go through a private
  `UnownedProperty()` helper.
- `OwnProperty(playerId)` — assigns owner, sets qualifier to
  `NeverBuiltOn`. Distinguishes transfer-between-players from first-time
  ownership: state only flips to `Owned` from `NotOwned` / `FreeParking`;
  transfers preserve `Mortgaged` / `Reserved` (the new owner inherits
  those states).
- `MortgageProperty()` / `ReserveProperty()` — guard against the wrong
  cross-state transition (mortgage refuses if reserved and vice versa)
  and silently no-op on unowned.

I'd initially flagged the silent no-ops as defects (under my outdated
read of the engine policy). Withdrew once the read-vs-mutation split
landed in the memory.

Rent-level normalisation: John's plan is for a separate
`PropertyService.NormaliseRentLevels(playerId)` to recompute rent levels
after any ownership-changing event, so the model mutators don't have to
juggle it. The `RentLevel = SINGLE` reset will move out of `OwnProperty`
once that exists.

## 8. `TransactionService` — first pass and rewrite

John's first pass landed (saved as `TransactionService_WIP.cs`) and
revealed how much rules-context the money layer actually carries. Real
defects found in review:

- **`ProcessTransaction` sign math.** The `amount > 0` branch inverted
  both balance updates — `ReceiveGoBonus` would *debit* the player.
- **`TakeoutLoan` passed `-amount`** for a *receive*; `RepayLoan` passed
  `+amount` for a *pay*. Caller-site inversions, partly cancelled by
  the `ProcessTransaction` bug above, mostly not.
- **Three receipt-sign inversions** (`PayJailFee`, `TakeoutLoan`,
  `RepayLoan`) — positive `Amount` on a paid transaction is the
  opposite of `event-receipts.md` §4 rule 1.
- **`PayRent`** didn't gate on the owner being in jail (Default rule 2).
- **`TryPurchaseProperty`** didn't gate on `HasPassedInitialGo`
  (Default rule 1).

John renamed the WIP file and asked for a fresh version. Wrote
`TransactionService.cs` from scratch around four decisions:

1. **One public method per `FinancialReason`.** Method names encode
   direction (`Pay…`, `Receive…`, `Take…`, `Process…`). Call sites read
   without thinking about signs.
2. **One core `Move` helper** with signed `long amount` and the same
   formula everywhere — `player.Money += amount`, counterparty player
   gets `- amount`, FP pot does the same algebra. The role-swap dance is
   gone.
3. **`allowShortfall` as caller policy** — true for debts (rent, tax,
   jail fee, mortgage fee, repay-loan, card charge), false for
   discretionary spending the caller must pre-gate (purchase, auction,
   build, unmortgage — Default rule 7).
4. **`ShortfallOutcome` as a tri-state** — `FundsRaised` (continue),
   `DebtSettled` (creditor-deal discharged the debt, skip the outer
   transaction), `Bankrupted` (skip the outer transaction). A boolean
   can't represent the `DebtSettled` case, which is exactly the path a
   settling deal needs (Default rule 7's "deal itself discharges the
   debt").

Dropped `Fine` from `FinancialReason` — every fine-shaped payment maps
to an existing reason (`CardCharge`, `JailFee`, `Tax`). No movement
needs a separate axis to group by.

## 9. Quirk: the subject is always the payer

The question John surfaced after reading the rewrite: "what if
counterparty player cannot pay — e.g., card says everyone pays you £25 —
do we call from the receiver's POV or the payer's?"

Worked out: **caller flips to the payer's POV, always.** Because `Move`'s
shortfall branch only fires for the subject (`if (amount < 0)`), the
subject must be the one who could fail to pay. Calling from the
receiver's POV silently underflows the counterparty's `Money`.

For "Happy Birthday — everyone pays you £25" the caller loops over the
payers:

```csharp
foreach (var payer in engine.Cache.Game.GetActivePlayers(excludePovPlayer: true))
    await _transactions.PayCardCharge(engine, payer, 25,
        TransactionCounterparty.Player, birthdayPlayer, ct);
```

Each payer's shortfall path is independent. The birthday player's
receipt stream is N positive `+25` entries — exactly the right
narrative. A bankrupt payer doesn't break the others; the recipient
just ends up short by that one payer's amount, which is correct game
state. (Whether the bank covers the shortfall is a `BankruptcyService`
decision; `game-rules.md` Bankruptcy rule 3 only guarantees that for
rent and fines.)

## 11. Architecture observations carried forward

- **The three-layer cut is the established pattern for rule services.**
  Pure data tables (`DoubleEffects`) describe *what the rule says*;
  primitive services do *one job each*; an orchestrator composes. New
  rules should fit this shape before reaching for new abstractions.
- **The engine error policy is two-mode, not one.** Reads bubble; rule
  mutations silently refuse. The web rolls back via snapshot reload.
  Apply this when reviewing engine code — silent returns in mutation
  methods are correct.
- **`TransactionService` is the single seam for money.** Every
  `player.Money` change goes through here; every
  `FinancialTransactionReceipt` is emitted here. The receipt stream is
  the authoritative narrative of money flow.
- **Payer-POV is the convention for player-to-player flows.** The
  caller iterates over potential payers, not over potential receivers.
- **`StreetRuleQualifier` is per-property but the rule is per-street.**
  Property-service mutations need to update the whole street as a unit
  when ownership events fire — single-property qualifier flips will
  leave the street in inconsistent state.
