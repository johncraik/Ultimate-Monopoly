## 5. The TransactionService ↔ shortfall DI cycle

The session's main architecture problem: shortfall resolution must dispatch to
Loan/Mortgage/Sell/Deal/Bankruptcy, all of which need `TransactionService`,
which needs *them* → a constructor cycle (same family as the 06-02 BoardService
cycle). Worked through several options:
- John's `LoanServiceBase` manual-construction (`new LoanServiceBase(this)`).
- My `IShortfallResolver`-on-the-bundle seam.
- John's `IServiceProvider`-in-the-bundle idea (I argued against — hides the
  graph, drags a container into the pure/testable engine, scope footgun, and the
  cycle doesn't actually need it).
- A static-helper extraction (`PropertyHelper.MortgageProperty` already done that
  way).

**Landing:** a `ShortfallService` (DI, ctor-injects `PropertyService` +
`LoanService`) injected into `GameEngineFactory` and passed into the
`GameEngine` bundle as `IShortfallService`. `TransactionService` stays
ctor-empty and reaches it via `engine.ShortfallService.ResolveShortfall(...)` —
no ctor dep, so the graph is acyclic under **plain constructor injection**, no
service-locator. `ResolveShortfall` (prompt + dispatch) moved out of
`TransactionService` into `ShortfallService`; `ShortfallOutcome` made public.
John's realisation that closed it: "we can just inject `ShortfallService` into
`GameEngineFactory`."

## 11. Shortfall coverage bug (found + fixed)

After the loop-termination rework, `ResolveShortfall` tracked coverage with
`remainingShortfall -= player.Money` — `player.Money` is the *whole* current
balance, not the amount raised, so a mortgage/sell that **over-covers** the debt
**uint-underflows** `remainingShortfall` → infinite re-prompt, and multiple
iterations double-count. Flagged it should recompute from the fixed target
(`while (player.Money < shortfallAmount)` / `remaining = max(0, shortfall -
Money)`), and pass `shortfallAmount` (not the running remaining) to
`ResolveViaLoan` so mortgage-then-loan doesn't double-discount. **John fixed it.**
