## 2. Per-game executor — built

The deadlock fix designed yesterday (`2026-05-27` §2) is now real.

- **`IGameExecutor` / `GameExecutor`** — a `ConcurrentDictionary` of
  per-game `GamePump`s. Each pump is an unbounded `Channel` drained by a
  single reader loop (`SingleReader = true`), so a game's work runs one
  item at a time on a background thread. That makes the cache's
  working-copy model single-writer *by construction*, not convention.
- **`GameWorkItem`** delegate — receives the resolved `GameEngine`
  bundle, the scoped `IServiceProvider` for that run, and the pump's
  cancellation token. `RunAsync` opens a DI scope, resolves the engine
  via `IGameEngineFactory`, runs the work, then pushes state on success.
- **Failure boundary** — a throwing work item is logged; the pump
  survives; the snapshot is the recovery boundary (there's a TODO to
  evict the dirty cache entry so the next item re-hydrates).
- **`SubmitPrompt` stays out-of-band** — it resolves the pending
  prompt's awaiter directly via `PromptProvider.TrySubmit` rather than
  queueing on the executor, so it unblocks the prompt-parked in-flight
  item instead of deadlocking behind it. `RunContinuationsAsynchronously`
  makes the resume hop onto the pump thread.

`GameService.EnqueueTurn` enqueues `StartPlayerTurn` + `ResolveThirdDie`
as the first consumer.

Two manual end-to-end runs passed: a plain roll+move, and a double-three
(extra turn). 110 engine tests still green.

## 3. `IEngineNotifier` — the engine→web seam

Added an abstraction in the engine (interface-in-engine / impl-in-web,
the same shape as `ISnapshotService`):

- **`IEngineNotifier`** — `PromptOpened`, `PromptClosed`,
  `StateChanged(GameCacheModel cache)`.
- **`PromptProvider`** now requires the notifier and fires
  `PromptOpened` + `StateChanged(cache)` after setting the pending
  prompt, and `PromptClosed` on clear. So a prompt opening mid-work
  pushes its own state frame even though the work item hasn't returned.
- **`SignalrEngineNotifier`** (web) broadcasts to the
  `game-play__{gameId}` group. `PromptOpenedMessage` / `PromptClosedMessage`
  records; the `Prompt` is typed as the base so the polymorphic `$type`
  serialises for the client.
- **`GamePlayHub`** gained `SubmitPrompt`, `GetCurrentPrompt` (covers the
  client that joins *after* the prompt opened — e.g. the first turn's
  dice prompt opens at game start, before phones navigate in), and
  `GetBoard` (the board is static for the game's life, fetched once
  rather than re-sent every frame).

## 4. Host-bypass — universal on the command gates

Per the Game-UI authority model (the host tablet is the authoritative
controller; phones are an optional convenience), the five turn-state
command gates now take `(string playerId, string submittingUserId)`:

```
CanPortfolioCommand, CanDeal, CanLeaveJail, CanEndTurn, CanDeclareBankruptcy
```

Each authorises "the named player **or** the host" via
`IsAuthorisedActor(playerId, submittingUserId) => submittingUserId == playerId
|| submittingUserId == cache.HostPlayerId`. Prompt responses already
enforced this in `PromptValidator`; the commands now match. This is the
engine change the host Play page depends on — "viewing a player profile
on the host should allow the same actions as if you were on your phone".

## 5. Live sync = push the cache, not receipts

A clear directive from John that reshaped the sync model: **the live
view broadcasts the whole `GameCacheModel` over SignalR; event receipts
are for history/stats, not for driving the live UI.**

- `StateChanged(cache)` fires at two moments: when a prompt opens (from
  the prompt seam) and when a work item completes successfully (from the
  pump's `RunAsync`). Between those, nothing — the cache is the frame.
- `GameCacheModel` got `[JsonIgnore]` on `Board` and `Events` — the
  board is static (fetched once via `GetBoard`), and the event/receipt
  stream is internal history, not part of the live frame.
- `ConcurrencyStamp` rides along as the version the client echoes back on
  `SubmitPrompt` for staleness detection.

Saved a memory ([[feedback_live_view_cache]]) because this directly
contradicts parts of `event-receipts.md` (§6/§8 read as if receipts
drive the live view) — those sections are now stale.

## 6. Prompt targeting — acknowledge prompts go to the affected player only

John clarified the acknowledge-prompt model: prompts that represent
something *happening to* a player fire only on **that** player's
profile. If player A collects rent from player B on B's turn, only B
gets the acknowledge prompt — A (the collector) does not. The engine is
a *helper that streamlines the UI/engine handshake*, not a simulator
that narrates to everyone. Added the targeting note to `choice-events.md`
(§5) and a short comment at the relevant prompt seam.

## 7. Board UI — the design decisions

Talked through the host Play page UI before building:

- **Prompt modals** — one partial per prompt type (not one mega-partial),
  surfaced on the player profile. The interruptible (NOPE) window is the
  one prompt that surfaces over the *game view*, host-authority.
- **Player profile on the host** — render the same `/player-profile`
  view inside a drawer (partial), **not** an iframe.
- **Board view** — fixed-size, responsive, *square*. Equal-sized space
  rectangles: top/bottom width = (W − 2·corner)/9, left/right height the
  same. Colour banner on the inward edge (top→bottom, bottom→top,
  left→right, right→left). Corners fill the 2×2; jail renders
  diagonally opposite Just Visiting; centre is the Free Parking pot.

These are written up in `web-orchestration.md` and the Game-UI doc.

with `BoardLayoutHelper` (the index→13×13-grid-cell mapping) and

The *process* was the problem. In order, the mistakes and corrections:

- **Banners on the wrong edge** — markup is banner-then-body; I set the
  flex directions as if it were body-first. Fixed.
- **Hardcoded hex colours** — "we have SCSS that ALREADY has property
  colours scaffolded through Bootstrap". Switched banners to the shared
  `.bg-prop-{slug}` palette via `PropertySet.ToDisplayName().ToSlug()`
  (e.g. DarkBlue → `bg-prop-dark-blue`, Station → `bg-prop-station`).
- **Recomputed model state** — "read BoardSpace and PropertyModel,
  you're duplicating helper methods/bools that already exist". Switched
  to `space.IsPurchasable` / `IsBuildable` / `IsTaxable` and JC.Core's
  `ToSlug`, rather than re-deriving them in the view.
- **Misdiagnosed the left-edge bug three times** (see §9).

The lesson, again: **read the classes I'm consuming before writing the
consumer.** Both corrections John gave me here are things a read-first

## 9. The unsolved left-edge bug

Symptom John reports: on the **left side only**, the station (15,
Marylebone) and utility (12, Electric) render with **no banner and no
hue** — while the colour properties on the left render fine, and the
stations/utilities on the other sides render fine.

I burned three wrong diagnoses on this:

1. "`IsPurchasable` excludes stations" — wrong; John: "is purchasable is
   correct… read it lol".
2. "stations/utilities have no `PurchaseCost`, use `IsRentable`" — wrong;
   they *do* have purchase costs, and this would affect *all* of them,
   not just the left.
3. "`ResolveSet` returns null for stations/utilities so `PropertySet` is
   null" — wrong; `PropertySetHelper.ResolveSet` correctly maps 12→Utility
   and 15→Station (`StationIndexes = [5,15,25,35]`,
   `UtilityIndexes = [12,28]`).

**What's been ruled out** (so future-me doesn't re-walk it):

- Board JSON data for 12 and 15 is correct (both have `SpaceType`,
  `PurchaseCost`, `Rents`).
- `IsPurchasable` is true for both → they take the ownable (banner+body)
  branch, not the text-label branch.
- `ResolveSet` assigns the right `PropertySet` for both.
- `BoardLayoutHelper.GetCell` left-edge geometry is symmetric with the
  (working) right edge; 12→row 10, 15→row 7, cols 1-2, no overlap with
  the centre or each other.

So the markup *should* be identical for a left station and a right
station. The bug is therefore **not** in the data, the helpers, or the
cell geometry as far as static reading goes — which means the next step
is to actually render it and inspect the emitted HTML for
`data-board-index="12"`/`"15"` (ownable branch? what classes on the
banner/body?), rather than reason about it further. Parked here at
John's call ("forget it").

Still also open: the **state hues are too dark** — `StateClass` returns
Bootstrap's `bg-success`/`bg-danger`/etc, and in the dark theme the
intended light tints (lime-green available, etc.) don't come through.
Wants light tints (e.g. `color-mix` toward white, ideally added to the
SCSS palette so the IDE resolves the theme vars).
