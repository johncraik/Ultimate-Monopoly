# 2026-06-02 -- Session 1 Notes

A long session that took the engine from "turn loop runs but landing does
nothing" to **roll → move → land → buy a property** working end-to-end. Along
the way: a comprehensive context reload, a `TransactionService` test suite that
surfaced a money bug, a sharp redefinition of *when* the cache commits, a
deep-dive into a production fault (the "gameId" executor faults), auctions
designed and built, the property-acquisition path (buy / auction / reserve)
fleshed out, a recurring `_working`/deep-copy class of bug chased down twice
more, and the first real in-game prompt partial.

Working pattern unchanged: John drives the implementation, I review / test /
flag / design. The persistent theme this session — flagged repeatedly — was the
**working-copy (`_working`) + `SaveChanges` + stale-reference** family of bugs.
Three separate bugs this session traced back to it. Lesson reinforced (twice):
when symptoms smell of "the mutation didn't stick," check the deep-copy ctor and
where `SaveChanges` fires *before* reasoning about the rule logic.

## 3. `TransactionService` test suite + the dice-bonus bug

Wrote `MP.GameEngine.Tests/ServiceTests/TransactionService_Tests.cs` covering
every public transaction type — debits, credits, player-to-player mirroring, the
Free Parking pot algebra, rent (board-driven), rounding, the shortfall flow
(prompt + abandon outcomes), and the no-op rules.

The reported dice-bonus bug (each payer lost £100, the roller only gained £100
instead of £100 + £100×N) root-caused to the **`_working` model**:
`TransactionService.Move` called `engine.Cache.SaveChanges()` at the end of every
call, which promotes `_working → _game` and nulls `_working`. The caller
(`ResolveDiceNumber`) held the roller reference from *before* that commit, so the
mirrored `+£100` credits landed on an orphaned copy and were lost on the next
commit. Same stale-reference-across-`SaveChanges` shape as a previously-fixed
bug.

## 4. `SaveChanges` belongs only at the turn-state boundary

The fix and the principle it forced out:

- John removed the per-`Move` `SaveChanges()` from `TransactionService`.
- Established the rule: **a turn's mutations accumulate on one `_working`
  instance and are committed exactly once, at the turn-state boundary.** Mid-turn
  commits detach any held model references.
- John then **moved `SaveChanges` into `GameCacheModel.SetTurnState`** so the
  commit rides on the phase change, and removed the explicit `SaveChanges` calls
  from `PlayerTurnOrchestrator` (`StartPlayerTurn` / `ResolveThirdDieMovement`).
- XML comments updated to match (`Move`, `SaveChanges`, `SetTurnState`); the
  stale `transactions.md` §7 ("SaveChanges happens mid-transaction") flagged as
  now-reversed.

A regression surfaced from this: `SetTurnState` delegated its re-stamp to
`SaveChanges`, which early-returns without stamping when `_working` is null — so
a pure phase transition stopped re-stamping the concurrency stamp. The
`TransitionToRollPhase_RestampsConcurrency` test caught it; guarded with an
`if (_working is not null) SaveChanges(); else StampConcurrency();`.

## 5. The "gameId" executor fault — board hydration is user-scoped

John reported `GameExecutor` broadcasting `GameFaulted`, the logged error always
being "no game cache for gameId / couldn't hydrate." Deep dive:

- The throw is `GameEngineFactory.GetAsync` when `GameCacheService.GetGame`
  returns null (cache miss **and** `HydrateAsync` returns null).
- Root cause: `HydrateAsync` resolved the board through the **user-scoped**
  `BoardCacheService`, but hydration runs in contexts without the board's owner —
  the **executor pump's background DI scope has no `HttpContext`, so no
  `IUserInfo`**. For a **custom-board** game the board lookup returned nothing →
  null board → null cache → the faulting throw. Default-board games were immune
  (default board is user-independent), which is why it was intermittent.
- John confirmed it himself by spotting every snapshot row had
  `CreatedById = "Unknown__Id"` (`IUserInfo.UNKNOWN_ID`) — the tell that the pump
  scope has no user.

John's fix: `HydrateAsync` resolves the board by **`game.UserId`** (the creator);
a new `BoardImportService.GetGameBoard` / `FilterBoardSkins(activeShareOnly:false)`
includes **revoked** shares so an in-play game survives the board being unshared
(the share-then-unshare-then-reload failure); `SnapshotService` writes snapshot
rows under `IUserInfo.SYSTEM_USER_ID`; the admin guard on `BoardCacheService.GetAllBoards`
was dropped (no longer a live exploit path after the change).

## 6. Game-board cache keyed by skin, not host

Review of the above surfaced a real defect: the new game-board cache was keyed by
**host user id**, so a host running two in-play games on two different custom
boards collided on one key (the second game's board served for the first on a
hydrate). John re-keyed it by **`game.BoardId`** (skin id) — content is
per-skin, not per-user — and aligned the entry's expiry to absolute. Agreed the
null-board case (skin genuinely deleted) caching a null is fine: there *is* no
board for the game, so failing every game on it is the honest outcome.

## 7. Turn-command gating — Layers 1 & 2

Discussed why a stray double-tap could fault the whole game: the capability gates
were advisory (checked on the connection thread, not re-checked on the pump), and
*any* work-item exception force-quit the table. Implemented:

- **Layer 1** — defensive no-op guards on the orchestrator entry points
  (`StartPlayerTurn` / `EndPlayerTurn` return early if the turn isn't in the
  expected state), mirroring the guard `ResolveThirdDieMovement` already had. An
  illegal/duplicate command now no-ops instead of throwing.
- **Layer 2** — the **authoritative gate re-check on the writer thread**:
  `GameService.EnqueueTurn` / `EnqueueEndTurn` now take `submittingUserId` and
  re-run `CanStartTurn` / `CanEndTurn` inside the work item before running the
  orchestrator. Callers thread the submitter (`GamePlayHub`, and `TryStartGame`
  passes `game.UserId` so the host's first-turn kickoff authorises).

**Parked for a later session (context for whoever picks it up):**
- **Layer 3** — whether a genuinely-unexpected exception should still force-quit
  the whole table via `GameFaulted`, or quietly evict + re-hydrate from the
  snapshot. Currently every throw force-quits; the snapshot is the recovery
  boundary, so most faults could recover silently and reserve the force-quit for
  true corruption.
- **Layer 4** — client-side: disable the Roll/End buttons on click and don't
  re-show Roll Dice while a prompt is open (reduces the double-tap surface;
  complements Layer 1/2, not a substitute).

## 8. Auctions — designed and built

Worked through the full flow, then wrote `design-docs/auction-flow.md`, updated
`choice-events.md` §15.8, added a `game-rules.md` **Auctions** section, and built
`AuctionService`. Key rule decisions (John's calls):

- **App-mediated bidding** (Option A), sequential `AuctionBidPrompt` per bidder.
- **Pass = out** (elimination); last bidder standing wins.
- **Minimum bid = 50% of cost = the reserve price**, grid-rounded
  (`MoneyHelper.MinAuctionBid`); the auction opens at this floor, never £0.
- **Affordability filtered up front** — only players who can afford the floor
  bid; the winner therefore can always pay, so there's no shortfall/bankruptcy
  path out of an auction, and no per-settlement edge case.
- **Last survivor is forced to take it at the floor** (a deliberate quirk); if
  the eligible list is empty (nobody affords the floor) the auction is cancelled
  and the landing is a no-op.
- **Bid by grid increment** — `MoneyHelper.AuctionIncrements(rule)` drives the
  raise buttons (`[50,100]` under "round to 50" up to `[1,5,10,20,50,100]` with
  no rounding), shipped on the prompt as `AllowedIncrements`; bids settle exact
  (no re-round). The winning bid *is* the price — no station scaling.
- **`AuctionService.RunAuction` returns an `AuctionOutcome`** rather than
  settling; `PropertyService` performs the acquisition (shared with outright
  buys). This also dropped `AuctionService`'s `TransactionService` dependency.

## 9. The DI cycle — landing resolution moved to the orchestrator

The `BoardService`/`PropertyService`/`FreeParkingService` stubs introduced a
**circular dependency**: `BoardService → JailService → MovementService →
BoardService` (the last edge from `MovePlayer` resolving the landing inline).
The .NET container throws on construction cycles, so the first turn would fault.
John cut it by **moving `ResolveBoardSpaceForPlayer` out of `MovementService` and
into the orchestrator** (called after each `MovePlayer`); `MovementService` now
depends only on `GoService`. Built and visualised the resulting graph — now an
acyclic DAG bottoming out at `TransactionService` / `DiceService`, with the
per-game `GameEngine` bundle correctly *outside* the scoped container (passed as
a method parameter).

## 10. Property acquisition + `NormaliseRentLevels`

The buy / auction / reserve landing path was built out in `PropertyService`. A
thorough review of `NormaliseRentLevels` found a cluster of real bugs, all fixed
by John via an **owner-based rewrite**:

- `GetOwnedPropertiesInSetByPlayer` computed the *current* player's holdings for
  every player key (wrong overload) → colour rents corrupted on every move.
- The station-count branch counted *players*, not the owner's stations → wrong
  level and a reachable `throw`.
- `Board.GetPropertySpaces` filtered with `IsProperty()` (buildable-only) →
  stations/utilities never normalised at all.
- `PropertySetHelper.MustReserve` had the arithmetic inverted (`== owned - 1`
  instead of `setSize - 1 == owned`) → reservation never triggered.

Rewrite keys each property's rent off **its own owner's** holdings
(`GetOwnedProperties(owner, set, includeMortgaged:false, includeReserved:false)`),
mapping a station's count to SINGLE/DOUBLE/TRIPLE/SET and a colour/utility to
SET-if-complete. `GetOwnedPropertiesInSetByPlayer` was removed. Confirmed the
station `throw` is now unreachable and the whole-board re-pass is the right
robustness choice (28-element loop; not a perf concern — John's point, and a
fair correction of an earlier nit of mine).

## 11. Mortgage rent flip + station purchase scaling

- **Rule flip (John's call):** a mortgaged property no longer counts toward its
  set for **rent** — the other members lose the set/station bonus until it's
  unmortgaged. Implemented as two coordinated changes (mortgaged → `SINGLE` in
  the top filter *and* `includeMortgaged:false` in the count); flagged that doing
  only the second would make the station branch throw.
- **Station purchase scaling** added on the outright-buy path (£200/250/300/400
  by stations already owned), and it **counts mortgaged stations** — deliberately,
  to close a loophole (buy → mortgage → buy-next-at-base would cap four stations
  at ~£840 vs the intended £1,150). This is a *deliberate* asymmetry with rent
  (which excludes mortgaged); documented in `game-rules.md` (Mortgaging rule 4 +
  Stations rule 2, cross-referenced) and a code comment, specifically so a future
  "consistency tidy" doesn't silently reopen the exploit.

## 14. The three movement / GO / rent bugs

End-of-session debugging, all three confirmed fixed (John can now roll, move
round the board, and buy properties — auction not exercised yet, no front-end
bid prompt):

- **Roller didn't move (but third die did).** `SetTurnDiceRoll` called
  `SetTurnState(PlayerRollMovement)` → `SaveChanges()` *mid-turn*, after the
  orchestrator had captured `player`; the next `cache.Game` access (the dice
  receipt emission) minted a fresh `_working`, detaching `player`. The move
  landed on the orphaned copy and was lost at the next commit; third die worked
  because its players are fetched *after* that commit. Fixed: removed the state
  change from `SetTurnDiceRoll` and added `TransitionToRollPhase()` at the top of
  `StartPlayerTurn` (before any `cache.Game` access, so it stamps without
  detaching).

- **Not-passed-GO didn't block buying.** Actual cause (John found it): the
  **`PlayerModel` copy constructor didn't copy `InitialRoll`** — every working-copy
  clone reset it to `false`, defeating the `!InitialRoll` guard in `MovePlayer`,
  so the *first* GO departure collected GO money and set `HasPassedInitialGo` on
  every player's opening move. Fixed by copying the field. (I'd flagged a
  *secondary*, dice-dependent path — a Double 3 returning the player to GO,
  clearing `InitialRoll`, then the extra turn re-departing GO — which John guarded
  separately: `InitialRoll` now only clears once the player has actually ended
  the move phase **off** GO: `player.InitialRoll && player.BoardIndex != GoSpace`.)

- **Rent of 0 still sent an acknowledge.** `PropertyService.PayPropertyRent` sent
  the "you owe £X" ack unconditionally before delegating to `PayRent`. Simple
  fix (John).

My miss worth recording: I traced the `InitialRoll` *logic* exhaustively but
assumed the clone preserved it — the bug was the missing copy-ctor field. Given
the `_working`/deep-copy theme, the copy ctor should have been the first place I
looked.

## 16. Architecture observations carried forward

- **The `_working`/`SaveChanges` model is sharp but unforgiving.** Three distinct
  bugs this session were the same shape: a model reference held across a
  `SaveChanges` (or a deep copy that dropped a field) silently loses mutations.
  The commit-only-at-turn-boundary rule is the structural defence; the deep-copy
  ctor is the other place this bites — keep both in mind on any
  "the change didn't stick" symptom.
- **Owner-keyed rule computations.** `NormaliseRentLevels` is correct now because
  each property's rent is derived from *its own owner's* holdings, not "any
  player" / "current player". Same principle will apply to building, deals, etc.
- **Deliberate asymmetries need a comment + a doc line**, or a later "tidy" will
  undo them (the mortgaged-counts-for-purchase-but-not-rent station rule).
- **Background (pump) scope has no user.** Anything resolved on the executor must
  not assume `IUserInfo`; resolve by stored ids (e.g. `game.UserId`) instead.
