## 3. 🌟 Deals — design (`game-deals.md`)

Worked through the whole deals model with John, then wrote
`design-docs/game-deals.md` (17 sections). Key decisions (John's calls):

- **Propose = command** (turn-boundary, `CanDeal`); **accept/decline = prompt**
  (`DealPrompt`) to the counter party. Only the counter party's accept/decline is
  a true engine prompt — the proposer always builds the deal and submits it whole.
- **Two entry points, one shared `RunDeal` core:** the turn-boundary command (free
  choice of counter party) and the **shortfall settlement** (counter party fixed
  to the creditor). An accepted shortfall deal **discharges the debt**
  (`ShortfallOutcome.DebtSettled` — the original rent is never charged).
- **No counter-offers** — decline ends it; renegotiate by starting fresh.
- **Dealable properties:** owned, not reserved, **mortgaged allowed**, NOT built-on
  (nor in a built-on set). This is `TradableProperties(includeMortgaged: true)`,
  *not* the FP-hand-in filter (which excludes mortgaged).
- **Money = cash on hand** (Default rule 7 — no raising funds for a deal).
- **Reserve rule:** don't block a deal while active; call
  `CheckReservationRuleSetObtained` for **both** players after accept.
- **Cursed deals allowed** — "I give £10, you give me everything"; not the app's job
  to police a bad deal a player agrees to.
- **Single-writer kills TOCTOU** — the game parks from propose through accept, so
  contents validated at build can't go stale.
- Chose **option #2 (a `BuildDealPrompt`)** for the shortfall builder over carrying
  the deal on `ShortfallResponse` — keeps the already-crowded shortfall prompt simple.

## 8. House/hotel inventory rules (build + sell) — multi-round review

A sequence of catches/fixes around the houses/hotels pool (`game-rules.md`
Building Rules; `RentLevel` order: SET, ONE→FOUR_HOUSES, HOTEL, DOUBLE_HOTEL):

- **Single-prop guard off-by-one + leaked into sell.** John added an inventory
  guard; it lived in the shared `RentLevelBuySellCheck` (so it blocked *selling*
  too) and used `HOTEL` as the house/hotel boundary (treating `FOUR_HOUSES→HOTEL`
  as needing a house, when it needs a *hotel* and frees 4 houses). John moved it
  into `CanIncreaseRentLevel` with the correct `[SET, FOUR_HOUSES)` / `[FOUR_HOUSES,
  DOUBLE_HOTEL)` split.
- **Bulk build freed-houses direction.** `CanIncreaseRentLevelForAllInSet` added
  `numHouses += numNewHotels*4` — adding *freed* houses to *demand*, which re-blocked
  hotel bulk-builds when houses were scarce. John fixed to the supply side:
  `numHouses > (housesLeft + extraHouses)`.
- **Sell IS inventory-gated** (I'd wrongly said it never is). Selling a `HOTEL →
  FOUR_HOUSES` *draws* 4 houses back. Confirmed correct: single-prop `HOTEL &&
  housesLeft < 4 → false`; bulk `housesRequired (4×hotels) ≤ housesLeft + numHouses`.
- **Underflow clamp.** John added a clamp in `GetHousesAndHotelsLeft` (clamp taken
  to the cap *before* subtracting) so over-allocation saturates `…Left` to 0 instead
  of `ushort`-underflowing to 65535 and defeating every inventory check.

## 10. 🌟 `GameCompletionService` (web) + runtime teardown

John built **`GameCompletionService : IGameCompletionService`**. `ConcludeGame`
loads the game, derives the outcome from the **cache** (>1 non-bankrupt → Drawn,
1 → Winner, 0 → throw), sets each `GamePlayer.PlayerGameOutcome`, `EndGame`, and in
one transaction updates the game + players + the `AppUser` win/draw/loss counts.

- **`_repos` vs `_context` shared context?** I worried the `AppUser` mutations
  (tracked on `_context`) might not be saved by `_repos.SaveChangesAsync()`. John:
  "read JC.Core docs" → `RepositoryManager` is a unit-of-work over **the underlying
  `DbContext`** (the same scoped `AppDbContext` instance), so the stats persist and
  are transactional. (Minor: the explicit `SaveChangesAsync()` before
  `CommitTransactionAsync()` double-flushes — harmless, matches `SnapshotService`.)
- **`ClearGameRuntime(gameId)`** (I added, at John's request) — evicts the cache
  (`GameCacheService.Invalidate`) + stops the pump. **Pump stop is fire-and-forget
  by necessity:** completion runs *on* that pump's work item, so awaiting
  `StopAsync` (which awaits the pump task) would **deadlock**. Called before the DB
  transaction (after the stat TODO). Bonus: the pump-stop CTS cancel also cuts the
  in-flight orchestrator tail (§9). Caveat: on a conclude rollback the runtime is
  torn down while the DB still says InPlay (error-path edge).
- John fixed `g.UserId` → `g.CreatedById` in the predicate (`UserId` is
  `[NotMapped]`, untranslatable).
- **PERF: cartesian explosion.** `ConcludeGame`'s query eager-`Include`d Players ×
  Turns × Snapshots × TurnEvents — a cartesian product where every row drags a
  duplicated `StateJson`/`EventJson` blob → hundreds of MB, "freezes / very slow".
  Fixed: include **only Players** (the stat projection loads the rest separately).
  Flagged the same milder pattern in `GameService.GetFinishedGame` (Players × Turns
  × BoardSkin.Spaces) — over-fetches, not fixed.
