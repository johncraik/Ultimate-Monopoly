## 1. 🌟 Issue Contact — the last C1 admin debt (yesterday's #1 open item)

The `/Admin/Logs/Issues/Contact/{issueId}` page (button existed, route 404'd) is now built.
- **`IssueContactService`** (`GetContext` + `SendContact`; ctor Admin/SystemAdmin/**GithubManager**). Composes
  the email (greeting → admin's message → the reporter's **own** original report → sign-off + reference),
  sends via `IEmailService` (auto-logged to `EmailLog`), writes an `AdminActionLog`. Re-checks contactability
  server-side (no stale-form send to a deleted reporter).
- **`UserManagementService.GetContactInfo`** → new lightweight `UserContactInfo(Email, DisplayName,
  EmailConfirmed)` record — **not** the full `UserViewModel` (no roles/2FA/lockout leak), so a GithubManager

  (DisplayName can be null **or** empty-string — `??` won't catch the latter; **John's note, memory saved**).
- **`GithubManagerPageFilter` widened** from the single `/Logs/Issues/Index` to a `/Logs/Issues/` **prefix**, so
  GithubManager can reach Contact.
- New `AdminActionType.IssueReporterContacted` + `AdminTargetType.Issue` (both appended — int-persisted) +
  `AdminLogService.LogIssueReporterContacted`.
- **Duplicate-contact warning:** `GetContactHistory` reads prior `IssueReporterContacted` admin-log entries for
  the issue and the page warns "already contacted N times, most recently … by …" (soft, not a block).
  **Retention dependency:** the (future) admin-log retention job must **not** prune `IssueReporterContacted`
  entries (they back the warning) — noted in code + a memory.
- **Email body** uses inline styles (email clients strip classes/CSS-vars — an intentional exception to the
  colour-variable rule, like the bug-report HTML). Delivers for real only post-**A1** (Console provider in dev).

## 5. 🌟 The four game-settings retention jobs

All recurring Hangfire, reading `GameSettings` via `SettingsDictionary`, idempotent, per-game `try/catch`,
System-attributed, `ILogger` summaries. Staggered UK crons.
- **`GameCleanupJob`** (03:00) — the **terminal hard-purge**. *Started* games-only (Option A), then **broadened**
  (John's call) to hard-delete **any** soft-deleted history record — `GameSnapshot`/`GameTurnEvents`/`GameTurn`
  by its **own** `DeletedUtc` past `CleanupRetentionMonths`, in FK-safe batches (leaves before turns). Closes the
  loop with the snapshot job. `Game`/`GamePlayer` shells are **kept** (FK-locked by active `PlayerGameStat`).
- **`GameAbandonmentJob`** (04:00) — `InPlay` games with no turn for `AbandonedRetentionWeeks` → **Cancel**
  (`TryCancelGame`) or **Draw** (`AdminGameStateService.BuildEngine` + `IGameCompletionService.TryDrawGameByAdmin`).
- **`CancelledGameCleanupJob`** (03:30) — soft-deletes `Cancelled` games last touched
  (`LastModifiedUtc`) past `AutoDeleteCancelledRetentionMonths`, **reusing `GameService.TryDeleteGame(isAdmin)`**.
- **`SnapshotCleanupJob`** (04:30, off by default) — **soft**-deletes snapshots+events of Finished/Cancelled
  games past retention; **stats-guard** (a Finished game is only purged once its `PlayerGameStat` exists, so the
  projection's source isn't destroyed first); the hard-reclaim is `GameCleanupJob`'s job.
- **`AdminGameStateService` refactor:** ctor guard **removed** — `BuildEngine` (read-only) is now open to any
  caller (jobs included); the destructive `TryRevertToTurn` keeps a **method-level** SystemAdmin guard.

**Key decision (the FK saga):** keep `PlayerGameStat` → so `Game`/`GamePlayer` shells linger as purged
tombstones. Chosen over dropping `PlayerGameStat`'s FKs (which would ripple into `GameStatsService` — it orders
the lifetime aggregate by `Game.CreatedUtc` and `.Include`s the navs — a stats-subsystem refactor).
