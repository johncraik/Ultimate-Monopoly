## 1. 🌟 The 11 bugs — root cause → fix

Grouped by the underlying defect, not John's numbering. **`§n`** = John's bug number.

### Cluster A — `SuppressDefault` not authored / not consumed (§3, §5, §7, §9)
The recurring shape: a card *does its own work* but the JSON **forgot the suppress flag**, so the
engine's default also runs (double-pay / double-increment). The call-site machinery was already correct.

- **§5 Cancel-triple-bonus double-incremented the accumulator.** `ApplyTripleBonus` *always* bumps the
  accumulator (`PlayerService.cs:130`, even on `factor:0`); the cancel card bumped it, then the
  orchestrator's default credit (`PlayerTurnOrchestrator.cs:274` `if (!sd.SuppressTripleBonus)`) bumped
  it **again** because the card carried **no `SuppressDefault`** and returned `None`. Fix: added
  `SuppressDefault{SuppressTripleBonus}` to the `third.json` card (mirrors the Triple-deck cards that
  already had it). Now the orchestrator skips its credit; only the card's `factor:0` bump remains →
  single increment, payout withheld. **John's "thread SuppressDefault into `CardActionContext`"
  hypothesis was the wrong shape** — the lost signal is *outbound* (card→orchestrator), and that channel
  (the returned `SuppressDefault`) already exists; it was just empty.
- **§9 Snake-eyes redirect paid from the holder, wrong order.** Card was `Direction=Pay, Target=Self,
  Counterparty=DiceOffPlayer` with **no suppress** → `OnSnakeEyes` fires in `DiceService.cs:43` (during
  the roll) **before** the £500 is credited in `HandleDoubleRoll`, so the holder paid £500 from their
  own balance *first* (transient −£500, shortfall risk). Fix: re-modelled to `Direction=Receive,
  Counterparty=Bank, Target=DiceOffPlayer, Basis=SnakeEyesBonus` + `SuppressDefault{SuppressSnakeEyes}`.
  The lowest roller now receives £500 from the **bank** directly; the holder's bonus is suppressed
  (`PlayerTurnOrchestrator.cs:234` already honours the flag). Holder's balance is never touched.
- **§7 "Receive ALL the FP money" then charged a fine.** The card empties the pot, control returns to
  `ProcessFreeParking`, the now-empty pot trips path-A (empty-pot pays the dice-difference fine).
  **John had already fixed the service side** (`FreeParkingService.cs:74/77` now read the aggregate
  `sd`, not the draw-only `suppressDefault`). Remaining piece (mine): added
  `SuppressDefault{SuppressFreeParkingMoneyTake}` to the `double.json` card — that makes the path-A
  guard `!sd.SuppressFreeParkingMoneyTake` false → the fine branch is skipped entirely.
- **§3 Anti-clockwise GO cards double-paid.** "Receive £500/£200 passing GO anti-clockwise, valid N"
  paid the card amount **and** the default £100 anti-clockwise pass bonus. John's call: they should
  **replace** the default (suppress it). Fix: added `SuppressDefault{SuppressGoBonus}` to both `go.json`
  cards — `GoService.CollectGoMoney` already gates the default on `!sd.SuppressGoBonus`. *(This overturned
  the `card-triggers.md` §9 "additive by design" table for these two cards — note for a doc sweep.)*

### Cluster B — Jail gating (§2, §8) — one root, two symptoms
`CardTriggerService` never checked the **holder's** jail state (only the *subject's*, via the
`JailFilter` switch). So a jailed bystander's hand was scanned and prompted for any-player triggers, and
`OnSpaceLand` (fired unconditionally in `BoardService.cs:36`) swept jailed bystanders during others'
third-die movement. Fix (one guard in `MatchingCardForTrigger`):
```csharp
where !player.IsInJail || trigger == CardTrigger.OnInJail
```
Jailed holders are excluded from **every trigger except `OnInJail`** — so get-out-of-jail-free **and**
befriend-a-guard (both ride `OnInJail`, the three `JailService` fire sites at `:75/:100/:140`) still
work, but jailed players no longer react as bystanders or play anytime cards. *(John's instruction named
only get-out-of-jail-free; the `OnInJail` carve-out is what keeps befriend-a-guard alive — it's a
`ModifyLeaveFee` card with no command path.)*

### Cluster C — Purge target (§4, §6)
- **§6 Opponent-purge prompted the opponent.** `PurgeProperty` was shared by self- and others-purge and
  conflated *chooser* with *owner* — the `TargetPropertyPrompt.PlayerId` was `purgingPlayer` (the
  opponent), so the opponent picked which of their own props to lose. Fix: split the signature to
  `PurgeProperty(chooser, owner, …)`. Prompt `PlayerId = chooser` (holder), eligible set = `owner`'s
  `BuiltOnProperties`, receipt attributed to `owner`. `PurgeOwnProperty` → `(player, player)`;
  `PurgeOthersProperty` → `(holder, opponent)`.
- **§4 "Friend purges 2 props" hit the cardholder.** `goToJail.json` Building action had no `Target`
  (defaulted `Self` → `PurgeOwnProperty`). Fix: added `Target:1` (ChosenPlayer). **John's call:
  single chosen opponent** (`Count:2` of one opponent — the existing `PurgeOthersProperty` path), not
  any-2-across-players (which would have needed new engine support). Works correctly now *only because*
  §6 is fixed (the holder picks the 2 props).

### Cluster D — Standalone (§1, §10, §11)
- **§1 "Mishandled evidence → Just Visiting" skipped pass-GO + buy-unlock.** `MovementActionService`'s
  `GoToJustVisiting` case hard-coded `CounterDirectionOfTravel`, which gates *both* the GO money and the
  `HasPassedInitialGo` unlock in `AdvancePlayer`. Fix: it now uses `MovementDirection(action)` (honours
  `CollectGoBonus`). Default `true` → `DirectionOfTravel` → a 30→10 clockwise crossing collects £200 and
  sets `HasPassedInitialGo` (unlocks buying, `BoardService.cs:47`). **Mass breakout** (`percentComChest`,
  jailed escapees) set `CollectGoBonus:false` to preserve no-collect; **John set call-a-meeting**
  (`justVisiting`) `CollectGoBonus:true` (advancing players *do* collect). Mishandled + Wrongful arrest
  keep the default (collect + unlock).
- **§10 Multi-use cards flip-flopped (200→500→200→500).** `CardService.PlayCard` re-added a kept/multi-use
  card via `.Add()` (end of hand). Fix: `Insert(0, card)` at both re-add sites — cards are drawn from the
  front, so a returned card goes back to the front and a multi-use card stays the *first* match next
  trigger. **Plus the real one-per-pass fix (John):** a one-card-per-player-per-trigger guard in
  `EvaluatePlayableCards` — the `playedCards` filter now also drops any candidate whose **holder** already
  played this resolution. (John first wrote it as `subject.PlayerId`; corrected to `c.Player.PlayerId` —
  `subject` is constant across the resolution, so using it kills *all* chaining the moment the subject
  plays, and never blocks a non-subject's 2nd card.) Net: the first card fires each pass until its
  `TurnsRemaining` hits 0 and it leaves the hand, then the second takes over; cross-player chaining intact.
- **§11 Loan repay cleared the loan *before* debiting → 4th-loan exploit.** `loan.PaidBack += outstanding`
  ran before `RepayLoan`, so during the repayment shortfall the loan looked settled and `CanTakeLoan`
  (`< 3 outstanding`) wrongly allowed a 4th loan to fund clearing loan #1. Fix: **reverse the order at all
  three sites** — `LoansActionService.RepayAll`, `LoanService.ForcedRepayLoans` (GO-pass instalment),
  `RepayLoansCustom`. Debit first → the loan stays outstanding during the shortfall → the max-3 gate holds
  → the player bankrupts instead of borrowing.

## 3. 🔴 Standing risk — the suppress-flag is hand-authored, with no guard
Bugs §5 and §9 were both **a forgotten `SuppressDefault` flag in JSON**. Correctness of every
"card does its own work + suppress the default" pair depends on the author hand-writing the right flag,
with **no engine-level guarantee** and no test. This is the brittleness the diagnosis kept surfacing.
Candidate next step: a lockstep/invariant check (every card whose action *replaces* a default carries the
matching suppress; every suppress flag has a consumer) — the `rule-citation.md` §10 pattern applied to
cards.

## 4. 🌟 CardImmunityService — first cut (built, NOT wired)
John's first pass at the Immunity substrate — **hardcoded immunity types**, the simplest viable shape
and the right call: it fits the engine's closed-finite-set constraint (`cards-design.md` §1/§3) and is
deliberately *not* the generic NOPE machine. Reviewed this session; sound and idiomatic.

- **`CardImmunity` enum** (5 types): `SwappingMoney`, `GoToJailCard`, `CancelledTripleBonus`,
  `PurgedProperty`, `ReturningProperty` — 1:1 with the 5 deferred immunity cards (GO / JV / FP×2 / GoToJail).
- **`ImmunityAction : CardAction`** carries a `CardImmunity Immunity` field.
- **`CardImmunityService`** mirrors `CardTriggerService`: 5 public `Check*Immunity(engine, subject, ct)`
  wrappers → a private `CheckImmunity` that scans the **subject's** hand for a matching immunity card,
  offers a single-option `CardOptionPrompt` (`PlayCardChoice` → decline = empty key), plays it on accept
  (consumed by `PlayCard`), and returns **`bool`** ("was it countered?"). The redundant double-lookup was
  simplified to one pass.
- Registered in DI (`ServiceRegistration.cs:50`, `TryAddScoped`). `CardService.ApplyAction` treats
  `ImmunityAction` as a no-op `true` (`CardService.cs:247`) — *holding* the card counters, *playing* it
  just consumes it.

**Not wired yet (deliberate — this is the cut, not the integration):**
- **No call sites** — the 5 `Check*` methods are invoked nowhere; the feature is inert until hooked.
- **Not exposed on the `GameEngine` bundle** (only `CardService` is; `CardTriggerService` is ctor-injected
  into the sub-services that need it). The hooks will reach it one of those two ways.
- **`[JsonDerivedType(typeof(ImmunityAction), "Immunity")]` NOT YET added** to `CardAction` — the import
  throws on any `"$type": "Immunity"` card until it is. **Top of the wiring list** (the only hard blocker).

**Wiring notes for when it's hooked (agreed in review):**
- Pass the **impacted** player as `subject` (the swap target / purged owner / bonus-cancel victim), never
  the holder — the service scans the subject's hand.
- Four hooks are symmetric ("an opponent's action aimed at you"), each skipping its effect on `true`:
  `SwappingMoney` → `MoneyActionService.SwapCash`; `CancelledTripleBonus` → `DiceActionService.ModifyTripleBonus`;
  `PurgedProperty` → `PurgingService.PurgeOthersProperty`; `ReturningProperty` → `PropertyActionService`
  return-to-bank.
- **`GoToJailCard` is the odd one** — it counters a *drawn* card, not a held opponent action, so its hook
  is at the GoToJail draw/resolve (`JailService.GoToJail`) and cancels the whole resolution.
- Immunity cards must be authored **kept-in-hand (non-`None` ConditionType) with NO `CardTrigger`** — else
  `CardTriggerService` would also surface them (double-fire). The immunity check is their only play path.
- **Jailed players can play immunity** — this path bypasses the trigger pipeline, so the §1 jail filter
  doesn't apply. Defensible (self-defence), but a conscious inconsistency with "out of jail only";
  flagged for a decision.
- **Single-use is baked in** (PlayCard consumes it) — correct for "play once".
- **Not NOPE.** A one-shot reactive counter; NOPE proper (universal, chainable, the `InterruptibleWindow`,
  `cards-design.md` §6) stays a separate, larger piece — don't bend this to cover it.
