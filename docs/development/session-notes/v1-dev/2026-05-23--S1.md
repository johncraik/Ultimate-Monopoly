## 2. The constraint that made the framework tractable

Key decision **stated by John, not the assistant**: a pending prompt does
**not** need to survive a server restart — only the `GameModel` snapshot
inside `GameCacheModel` does. This single constraint unlocks the entire
framework:

- The engine can `await` a response inline using `TaskCompletionSource`,
  rather than serialising a continuation or modelling each prompt as a
  separate command in a phase machine.
- A turn interrupted by a restart is lost — the player whose turn it was
  re-enters their roll. The previous turn's snapshot is intact, so no
  committed state is at risk.
- Replay / backtrack continue to operate on the snapshot timeline; prompts
  are *inputs* to a turn, not part of its permanent record.

The async/TCS path would have been hard to justify if mid-turn persistence
were required; with the restart contract above, it's the right choice.

## 3. Rename: Choice → Prompt

The first design draft used `Choice<TResponse>` everywhere. John pushed back
mid-design — the mechanism isn't choice-specific. An acknowledgement ("you
can't afford this, auction begins — OK?") is the same pause-and-await
pattern but has no decision to make. A dice input is the same pattern with a
typed value. Naming should reflect the *mechanism*, not one of its
applications.

Renamed everything top-to-bottom — `Choice` → `Prompt`,
`IChoiceProvider` → `IPromptProvider`, `PendingChoice` → `PendingPrompt`,
SignalR events `PromptOpened` / `PromptClosed` / `SubmitPrompt`. The doc
now names four shapes — Acknowledgement, Choice, Input, Interruptible
Window — all using the same `Prompt<TResponse>` / `PromptResponse` types.

(Filename `choice-events.md` kept; a note at the top of the doc explains
the legacy name.)

## 4. Interruptible window pattern — host Continue, not timeouts

Discussed how to handle a card play that another player could NOPE:

- **Timeout-based** — wait N seconds for NOPEs to come in. Rejected: imposes
  artificial pressure, depends on every phone being connected.
- **Per-player poll** — ask each NOPE-holder "do you want to NOPE?".
  Rejected: forces non-holders to also tap something, friction with no
  purpose.
- **Host Continue** (chosen) — the engine pauses; the host (tablet
  controller) taps Continue when the table is done discussing. Matches the
  "helper, not simulator" principle from `game-engine.md` §1 — the NOPE chat
  happens around the physical table, the app just records the outcome.

The pattern generalises beyond NOPE — any card-interrupt scenario. One
prompt (`InterruptibleWindowPrompt`), two response paths
(`Continue` from the host only, `PlayCard` from any eligible holder), no
timeout, the table decides when to close.

**Chain implementation:** a `while (true)` in the engine, opening a new
window each iteration if eligible responders remain. C#'s `async/await`
state machine handles the continuation — no explicit chain bookkeeping.

## 5. Framework implementation (`MP.GameEngine`)

Built the framework in:

- **`Models/Prompts/`** — `Prompt`, `Prompt<TResponse>`, `PromptResponse`,
  `PromptTarget` (+ `PromptTargetKind` enum), `PendingPrompt`.
- **`Models/Prompts/PromptTypes/`** — concrete prompts (added later).
- **`Models/Prompts/PromptTypes/Responses/`** — concrete responses (John
  later split responses into this subfolder).
- **`Abstractions/IPromptProvider.cs`** — the seam between engine and outside
  world.
- **`Services/Framework/`** — `PromptProvider` (default `IPromptProvider`)
  and `PromptValidator` (static, type-switched).

`GameCacheModel` gained `PendingPrompt? PendingPrompt`,
`SetPendingPrompt(PendingPrompt)`, and `ClearPendingPrompt()` — both
mutators re-stamp `ConcurrencyStamp`. The pending slot is singular by
design: `SetPendingPrompt` throws if one is already pending.

`PendingPrompt` is a public class whose `TaskCompletionSource<PromptResponse>`
is `internal` — clients can read the public `Prompt` for SignalR rendering
but cannot touch the resolution mechanism.

`PromptProvider.RequestAsync` uses `await using (ct.Register(...))` so
cancellation cleanly propagates out of any awaiting engine code.

## 6. Concrete prompt types — implemented in order

### 6.1 `InterruptibleWindowPrompt`

The card-interrupt pattern from §4 above. Carries `EligiblePlays`
(`(PlayerId, CardId, CardName)` triplets), no `Timeout` ever, response is
`Continue` | `PlayCard`. Validator allows `Continue` only from the host;
`PlayCard` must match an entry in `EligiblePlays`, submitter must be the
named player or the host.

**Important boundary clarified during discussion:** the validator does
*not* remove the played card from the player's hand. State mutation —
removing the card, discarding it, emitting `CardPlayedReceipt` — is engine
code that runs *after* the prompt resolves. The framework's job is
pause + validate; the engine handles consequence. Single-prompt-slot
invariant guarantees no race between prompt opening and resolution.

### 6.2 `AcknowledgePrompt`

Fancy notification — single OK button, empty response, pauses the engine
until acknowledged. Used when the rules force an outcome the player should
be told about ("you can't afford this — auction begins"). `Timeout` +
`DefaultResponse` set per call site for auto-dismiss when desired.

### 6.3 `DiceRollPrompt`

Input prompt for entering dice values. Covers the standard 3-dice turn
roll and card-forced rolls of 1, 2, or 3 dice. Setup-phase dice are out of
scope (they go through the setup hub).

**Design decisions:**

- **One prompt with `DiceCount`** (`ushort`) over three separate types or
  a flat `IReadOnlyList<uint>`.
- **Response is named fields, not a list** — `Die1: ushort`,
  `Die2: ushort?`, `ThirdDie: ushort?`. John pushed back on the
  flat-list approach: it requires a convention ("`Dice[2]` is the third
  die"), and conventions are bugs waiting to happen. Named fields force
  the consumer to explicitly state which is the third die — the rules
  themselves call it "the third die", and the code now mirrors that.
- **`ushort` for dice values and count** — domain-precise; 1–6 fits
  trivially, validator enforces range.

Validator enforces that exactly `DiceCount` of the three fields are
populated — no extras, no missing — and every populated value is in 1–6.

### 6.4 `AcquirePropertyPrompt`

Originally drafted as `BuyPropertyPrompt` with a 3-way action
(Buy / Reserve / Decline) and a `Mode` enum. John collapsed this:

> Reservation is, under the hood, just a "buy" with a special branch. The
> prompt's job is the binary yes/no — whether "yes" means buy or reserve
> is engine state, decided when the prompt is created and again when the
> response is handled.

Renamed `AcquirePropertyPrompt`, response is just `Accept: bool`.
Discriminator `AcquireProperty`.

**Other property-related decisions:**

- `PropertyId: string` → `BoardIndex: ushort` (the actual identity on
  `PropertyModel`; `PropertySetHelper.ResolveColour(ushort)` maps it to a
  set).
- Kept `Cost: uint` — John liked the idea that the frontend could compute
  "you'd spend 2% of your money" from structured data.
- Affordability is gated *before* the prompt opens: if the lander can't
  afford, the engine emits an `AcknowledgePrompt` and proceeds straight
  to auction. The prompt only opens when the offered action is genuinely
  affordable.

## 7. Metadata split — John's refactor

Mid-session, John refactored `GameModel`, `GameMetadata`, and
`GameCacheModel`. Two distinct concerns were tangled in the old
`GameMetadata`:

| Concern | Now lives in | Source | Lifetime |
|---|---|---|---|
| Game identity / config | `GameCacheModel` | constructed once from `GameDTO` | game-wide, effectively immutable |
| Turn state | `TurnMetadata` (inside `GameModel`) | mutated by the engine | per turn |

`GameId`, `GameName`, `BoardId`, `HostPlayerId`, `RoundingRule`,
`GameState`, `GameOutcome` → cache (from DTO).
`CurrentTurnId`, `CurrentPlayerId`, `TurnNumber` → `TurnMetadata`.

`GameMetadata` is gone; `TurnMetadata` is the new turn-scoped type.

**Why this matters:** snapshots no longer carry redundant identity data
(game id, name, board id, host id, rounding rule, lifecycle markers — all
stable for the entire game and already in the `Game` relational header
per `game-engine.md` §8 rule 3). Each per-turn JSON blob is now smaller
and purely about *what changed*.

The standout consequence for the prompt framework was the next item.

## 8. `HostPlayerId` removed from every prompt

Because `HostPlayerId` now lives on `GameCacheModel` (sourced from the
DTO, never changes during a game), it is **redundant** on every individual
prompt. The validator already receives the cache. So:

- Removed `HostPlayerId` from `InterruptibleWindowPrompt`,
  `AcknowledgePrompt`, `DiceRollPrompt`, `AcquirePropertyPrompt`.
- `InterruptibleWindowPrompt.Target` simplified — no longer appends the
  host id; the audience is just the eligible players. The host sees the
  prompt via group broadcast anyway, no need to be in `Target.PlayerIds`.
- Every `PromptValidator` branch now compares against `cache.HostPlayerId`
  instead of `prompt.HostPlayerId`.

Single source of truth, one less field on every prompt construction site,
no risk of an engine call site forgetting to set it.

## 11. Architecture observations worth carrying forward

- **Framework / engine separation is sharp.** The framework pauses and
  validates. The engine mutates state. Validators are pure read-only —
  if validation succeeds, the engine's `await` returns and the engine
  code does the consequence. This was tested by the "should the
  validator remove the card?" question — answer is firmly no.
- **Concurrency stamp does double duty.** Originally just for guarding
  stale commands; now also serves as the "first-responder-wins" mechanism
  for prompts open to multiple players. When the cache clears a prompt
  on submission, the stamp changes, and the second submitter against
  the same prompt loses by stamp mismatch. No special target kind
  needed.
- **Cache vs snapshot is the right axis.** Game-identity on cache,
  turn-state in snapshot. Anything that doesn't change during the game
  shouldn't be in the snapshot. Anything in-memory only (events,
  prompts) sits on the cache and dies on restart by design.
