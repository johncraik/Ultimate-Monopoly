## 1. The DI question — how do framework services get their cache?

Opened on the tension that should have been obvious from yesterday:
`PromptProvider`, `TurnStateProvider`, and `EventEmitter` all take
`GameCacheModel` as a constructor parameter, but `GameCacheModel` is
**per-game** (one instance per `gameId`, lives in `IMemoryCache`).
Scoped DI can't resolve it without knowing the `gameId`, which only
exists at runtime on a hub/controller call. As registered, the three
providers' scoped registrations would have thrown.

Three shapes considered:

1. **Per-game factory** — caller resolves an `IGameEngineFactory`,
   does `factory.GetAsync(gameId)`, gets back a small bundle holding
   the cache + three providers built around it.
2. **Child DI scope per game** — register `GameCacheModel` as scoped
   in a per-game child scope; providers resolve normally. Cleaner DI
   but heavy plumbing, no good middleware hook for SignalR hubs.
3. **Stateless providers** — take `cache` as a method parameter on
   every call. Simplest but throws away the "one provider per cache"
   framing the design docs already establish.

John picked **option 1** and built it.

## 2. `GameEngine` bundle + `IGameEngineFactory`

`MP.GameEngine.Services.Framework.GameEngine` — sealed class that
takes `(GameCacheModel cache, ISnapshotService snapshotService)` and
exposes `Cache`, `PromptProvider`, `TurnStateProvider`,
`EventEmitter`. Providers are `new`'d directly in the bundle's field
initialisers — fine for production, slight constraint on
substitution in unit tests (rule-service tests have to either mock
the whole bundle or stand up real providers around a test cache).

`MP.GameEngine.Abstractions.IGameEngineFactory` — single method
`Task<GameEngine> GetAsync(string gameId)`. Engine-side interface,
implementation lives in the WebProject (and isn't written yet —
deliberate, abstraction first). Same interface-in-engine /
impl-in-web pattern applies to `ISnapshotService`. Both interfaces
have a `//Implementation created in WebProject` comment that I
flagged as a minor "ages badly" smell.

The three previously-broken scoped registrations for
`IPromptProvider` and `IEventEmitter` were dropped from
`ServiceRegistration.AddGameEngine` — the factory is now the only
legitimate construction path.

## 3. The storage-boundary debate

I initially read the transitions taking `ISnapshotService` and
calling `snapshotService.TryCreate…Async` as a contradiction of
`game-engine.md` §3 ("engine knows nothing about storage"). John
pushed back, correctly:

> The engine still knows nothing about storage — it's calling an
> interface. That interface implementation could be storing DB
> records; could be writing files; or whatever. Engine is oblivious
> to how we snapshot — it just now knows that when we end a turn, it
> writes a snapshot.

Right. The storage *abstraction* sits at the boundary; the engine
depends on a contract, not a mechanism. The principle holds; what
changed is *when* a snapshot happens — now intrinsic to the
transition itself, consistent with yesterday's "mutations intrinsic
to a transition belong on the transition" principle.

`game-engine.md` §3 wording could be sharpened: not "engine knows
nothing about snapshots" but "engine doesn't know *how* snapshots
are persisted, only *when* one is taken".

## 4. Naming convention cleanup — `Try…` doesn't throw

The first version of `ISnapshotService` had `TryCreateFullSnapshotAsync`
and `TryCreatePartialSnapshotAsync` returning `bool` — but the
transitions awaited and discarded the result. Silent failure path.

Decision: throw on failure, drop the bool. Which made the `Try…`
prefix a lie — .NET convention is `Try…` = returns bool, never
throws. Renamed to `CreateFullSnapshotAsync` / `CreatePartialSnapshotAsync`.
This kind of thing matters: a future reader sees `await TryCreate…`
and assumes they should branch on a result that doesn't exist.

## 5. The partial-snapshot pivot — and the schema implication

`SnapshotService` implementation (web side) wrapped `BeginTransaction`
/ `Commit` / `Rollback` around the inserts, logged + rethrew on
failure — clean shape. But the partial-snapshot flow had a real bug:

`GameSnapshot.TurnId` is `[Key]` in the EF model (single-column PK).
Multiple partial snapshots within the same GameTurn would collide on
PK insert. This was the exact 1:many tension yesterday's notes had
flagged as "schema needs to change to support 1:many". Two options
to resolve:

1. **Compound key** `(TurnId, SequenceWithinTurn)` on GameSnapshot,
   migration to match.
2. **Drop the partial concept entirely** — every snapshot creates a
   new GameTurn + GameSnapshot pair. Extra-turn snapshots get a new
   GameTurn record with the *same* `CurrentPlayerId`.

John picked **option 2**. `ISnapshotService` collapsed to a single
`CreateSnapshotAsync` method. Both transitions now write a new
GameTurn + GameSnapshot pair; the only thing distinguishing extra-turn
from next-player is whether `CurrentPlayerId` advances.

This **resolves the schema tension cleanly** — 1:1 stands, no
migration needed, yesterday's "§8 contradiction" flag is moot.

### 5.1 The semantic shift on `TurnNumber`

Yesterday's design pinned "extra turn = same `TurnNumber`, same
player, new snapshot row". New design: `TurnNumber` bumps on every
transition (`UpdateMetadata` runs in both paths). So `TurnNumber`
now counts snapshot/turn-record units, not player-rounds.

Stats implication: "how many turns has Alice had?" now returns 3 if
she rolled doubles twice, not 1. If a stat needs "rounds completed",
it has to derive from `COUNT(DISTINCT CurrentPlayerId transitions)`,
not `MAX(TurnNumber)`. Not wrong, just a different definition than
yesterday's.

## 6. `CurrentTurnId` — the in-place mutation that almost wasn't

After the partial-snapshot collapse, a real gap emerged:
`SnapshotService` generates the new `GameTurn.Id` server-side
(default field initialiser `Guid.NewGuid().ToString()`) but didn't
write it back to `game.Metadata.CurrentTurnId`. The DB had a row the
cache didn't know about; any read of `cache.Game.Metadata.CurrentTurnId`
between transitions would return the *previous* turn's id.

I proposed a return-the-id refactor (snapshot service returns
`Task<string>`, engine writes the result to the cache). John picked
a different fix: have `SnapshotService` write `game.Metadata.CurrentTurnId
= turn.Id` directly on the passed-in `GameModel`. Works via
`GameCacheModel`'s working-copy semantics (the `Game` getter returns
the lazily-created `_working` reference; any mutation lands on
`_working`; the trailing `cache.SaveChanges()` promotes
`_working → _game`).

Trade-off: the in-place mutation crosses an abstraction boundary
silently — `CreateSnapshotAsync(GameModel game)` doesn't advertise
in its signature that it writes back. Resolved by **adding explicit
XML docs on the interface**:

> A new turn id is generated as part of this call and written back
> to `game.Metadata.CurrentTurnId` so the in-memory cache and the
> database agree on the current turn after the call returns —
> callers should expect `game.Metadata` to be updated.

John's framing was right: "snapshot creates a new turn; thus you
should expect turn metadata to update". The doc makes the contract
explicit.

## 8. `TryStartGame` — first real consumer of the framework

John added a `TryStartGame(string gameId)` to the web-side
`GameSetupService`, the first thing that actually composes the
foundation framework end-to-end. Shape:

1. Fetch the setup-state `Game` row, validate host is the caller,
   validate player count, validate every player has set their dice.
2. Build `GameDTO` and `PlayerDTO`s.
3. `_engineSetupService.SetupGameCache(...)` builds the
   `GameCacheModel` — engine side. (Engine setup service had to be
   updated to seed `TurnMetadata` — see below.)
4. `_snapshotService.CreateSnapshotAsync(cache.Game)` writes the
   baseline GameTurn + GameSnapshot row, mutates back `CurrentTurnId`.
5. `_gameCacheService.PopulateGame(cache)` stores the cache in
   `IMemoryCache`. Order matters — snapshot first, then populate,
   so concurrent reads can't see a cache with an empty
   `CurrentTurnId`.
6. `_gameCacheService.SaveChangesAsync(gameId)` promotes the
   `CurrentTurnId` write-back from `_working` to `_game`.
7. `game.StartGame()` flips the DB row from `GameState.Setup` to
   Active, persisted via `UpdateAsync`.
8. SignalR broadcast `"GameStarted"` to the setup-hub group.

### 8.1 Issues caught and fixed during review

- **`TurnMetadata` was never initialised** — the engine-side
  `GameSetupService.SetupGameCache` left `Metadata = null`. `CreateSnapshotAsync`
  would NRE on `game.Metadata.CurrentPlayerId`. Fixed by seeding
  `Metadata` with `TurnNumber = 1` and `CurrentPlayerId =
  playerDtos.MaxBy(p => p.Dice1 + p.Dice2)?.Id` (highest setup-dice
  total goes first, per game-rules.md initial roll-off).
- **No authorisation check** — any authenticated user could start
  any setup-state game by guessing the id. Fixed by adding
  `if (game.CreatedById != _userInfo.UserId) return false;` after
  the null check, matching the pattern in `TryKickPlayerFromGame`.
- **`Game.State` never flipped to Active** — the DB row stayed in
  Setup, meaning the same game could be "started" forever. Fixed
  via a new `game.StartGame()` entity method + `UpdateAsync`.
- **Order swap** — original code populated the cache *before*
  creating the snapshot, meaning a concurrent request could see a
  cache with no `CurrentTurnId`. Swapped to snapshot-then-populate.

### 8.2 Known-but-not-fixed-today

- **Transaction split.** `CreateSnapshotAsync` has its own internal
  tx; `game.StartGame()` + `UpdateAsync` is a separate save. If the
  second one fails, you have a snapshot row but the game is still
  marked Setup. Next `TryStartGame` would create *another* initial
  snapshot. For debugging it's fine; for production the snapshot
  insert + Game state flip should share an outer transaction.
- **Dice tie-breaking.** `MaxBy` silently picks one of N tied
  players. Real Monopoly requires ties to re-roll. Punted —
  not blocking the happy path.

## 10. Architecture observations worth carrying forward

- **The factory pattern (one bundle per game-id) is the right cut.**
  It matches what the provider doc comments already claimed ("one
  instance per game cache"), keeps providers' constructor shapes,
  and gives rule services / orchestrators a single obvious entry
  point. Per-game scopes / stateless providers were the alternatives;
  this is materially cleaner for SignalR-driven flows.
- **Interface-in-engine, impl-in-web is the right layering for
  storage and factory.** The engine references `ISnapshotService` and
  `IGameEngineFactory` only; both implementations live in the
  WebProject where they can touch EF / `IMemoryCache` /
  `IHubContext`. Engine assembly stays free of web/persistence deps.
- **The cache's working-copy semantics are load-bearing for
  mutate-then-promote flows.** `SnapshotService.CreateSnapshotAsync`
  writing back to `game.Metadata.CurrentTurnId` works *because*
  `cache.Game` returns the mutable working-copy reference. If anyone
  ever changes the getter to return a defensive copy or a read-only
  projection, this silently desynchronises the cache from the DB.
  Worth keeping in mind when refactoring `GameCacheModel`.
- **"`Try…` returns bool, never throws" is worth defending.** The
  snapshot service rename felt small but the principle is general:
  prefix-as-contract should be honoured or dropped.

## 11. After the break — `GameEngineFactory` + cache refactor

Picked back up by implementing `IGameEngineFactory`. The factory itself
was a four-line wrapper:

```csharp
var cache = await _cacheService.GetGame(gameId) ?? throw ...;
return new GameEngine(cache, _snapshotService);
```

But surfaced a circular dependency that had crept in earlier: someone
(me?) had changed `GameCacheService.GetGame` to call
`_gameEngineFactory.GetAsync(gameId)` on a cache miss. With the factory
also calling back to `GameCacheService`, a single cache miss would
infinite-loop (or deadlock on `IMemoryCache.GetOrCreateAsync`'s
re-entrant create lambda).

Resolved by reshaping the responsibilities cleanly:

- **`GameCacheService`** — owns the full cache lifecycle. Memory-cache
  + DB hydrate. Deps: `IMemoryCache`, `IRepositoryManager`,
  `BoardCacheService`, `GameEngineSetupService`. On miss: load the
  latest `GameSnapshot` for the gameId (joined to `GameTurn`, ordered
  by `TurnNumber` DESC), resolve the board, build the `GameDTO`, call
  `engineSetup.SetupGameCache(dto, json, board)`, store, return.
- **`GameEngineFactory`** — pure wrapper. Deps: `GameCacheService`,
  `ISnapshotService`. Wraps the cache in the provider bundle.

Net chain: `Factory → CacheService → { IMemoryCache, IRepositoryManager,
BoardCacheService, GameEngineSetupService }`. No cycle.

Hydration covers the two real scenarios: server restart with an
in-flight game, and a cache entry that expired (12h sliding window).
Setup-state games are rejected by the snapshot null-check — they have
no snapshot row to deserialise from, so the factory throws and the
caller deals.

## 12. Caller-driven snapshot transactions

The tx-split issue I'd flagged in §8.2 needed addressing — the original
`TryStartGame` ran `CreateSnapshotAsync`'s internal tx, then did a
separate save for the `Game.State` flip. Partial failure left orphan
snapshot rows.

Solution: extend `ISnapshotService.CreateSnapshotAsync` with
`bool completeTransaction = true`. `true` (default) keeps the original
behaviour — snapshot service begins/commits its own tx. `false` lets
the caller compose the snapshot insert into a larger outer tx.

Two correctness bugs surfaced during this pass and were fixed:

1. **Rollback was unconditional in the catch block.** When
   `completeTransaction == false`, the snapshot service would still
   roll back the caller's transaction on failure — and the caller's
   own catch would then roll back again. Fix: only rollback when this
   method owns the tx.
2. **First version swallowed the exception when `completeTransaction
   == false`.** Logged it and returned. Caller's later
   `SaveChangesAsync` would happily commit orphan GameTurn rows.
   *Worse* than the double-rollback. Fix: always rethrow; only the
   rollback is conditional. Contract is now "snapshot service surfaces
   failure via exception; whoever owns the tx owns the rollback".

## 13. `TryStartGame` ordering + tx wrap

With caller-driven tx support in place, `TryStartGame` rewired to:

```
game.StartGame()                                    // in-memory entity flip
build gameDto + playerDtos
cache = engineSetup.SetupGameCache(...)             // in-memory cache build
BeginTransactionAsync
try {
    snapshotService.CreateSnapshotAsync(cache.Game, completeTransaction: false)
    repos.GetRepository<Game>().UpdateAsync(game, saveNow: false)
    repos.SaveChangesAsync()
    repos.CommitTransactionAsync()
} catch {
    RollbackTransactionAsync + log + return false
}
cache.SaveChanges()                                 // promote CurrentTurnId write-back
gameCacheService.PopulateGame(cache)                // populate IMemoryCache
hub.SendAsync("GameStarted")                        // broadcast
```

The pre/post split is load-bearing: `IMemoryCache` is not
transactional, so cache population and the SignalR broadcast must
happen *after* `CommitTransactionAsync` — otherwise a rolled-back
start would leave a started-state cache + a `GameStarted` event for a
game that doesn't exist.

End state: snapshot insert + Game-state flip are now atomic. Either
both land or neither does; the cache and the broadcast only fire on
success.

## Architecture observations carried forward

- **The factory pattern bottoms out at a thin wrapper.** All real
  logic lives in `GameCacheService` (cache + hydrate) and the
  providers (transitions, prompts, emission). The factory is the
  composition seam, nothing more. Resist the temptation to put logic
  in it.
- **`IMemoryCache` is not transactional.** Any multi-write operation
  that touches both the DB and the cache must commit the DB before
  populating the cache. Same goes for broadcast — the SignalR push
  has no rollback either, so push after commit.
- **Working-copy semantics on `GameCacheModel` are load-bearing.**
  `cache.Game` returns the lazily-created working copy; mutations land
  there; `SaveChanges` promotes. Any consumer that mutates `cache.Game`
  is *implicitly* writing back to the cache. The snapshot service
  setting `game.Metadata.CurrentTurnId` works only because of this. If
  the getter ever changed to return defensive copies, half the
  framework would silently desynchronise.
- **"Try…" should mean "returns bool, never throws".** Honoured for
  the snapshot service rename. Reinforced this session by the
  `completeTransaction` failure-path bug — `Try…` implies the caller
  branches on a result, and swallowing exceptions to honour that
  contract is worse than just throwing. Naming should match exception
  semantics.
