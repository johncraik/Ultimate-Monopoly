## 2. Board skin editor — create vs. edit flow

**Files:** `Pages/Boards/Edit.cshtml`, `Pages/Boards/Edit.cshtml.cs`

Goal: when *creating* a new skin, the page should only show the Details card;
the Spaces customisation UI should appear only after the skin exists.

- Spaces card and its two modals (create/edit space, delete space) now render
  only in edit mode (`Model.Id is not null`).
- Buttons:
  - **Create mode** — a single **Next** button.
  - **Edit mode** — **Save** and **Save & Close**, both submitting an `Action`
    form value (`save` / `close`).
- `OnGetAsync` returns early for new skins, skipping the board/section build
  entirely (no wasted work).
- `OnPostAsync` gained an `Action` parameter: `"close"` redirects to the boards
  index; otherwise it reloads the page in edit mode (`?id=...`). The **Next**
  button follows that same reload-into-edit-mode path after the skin is created.

**Decision:** reused the existing post-redirect-get behaviour — creating a skin
already redirected to `?id={newId}`, which naturally becomes the "reload to show
spaces" step, so no extra plumbing was needed.

## 6. Editor default-name bug fix

**File:** `Pages/Boards/Edit.cshtml.cs`

John reported the editor's "default name" was wrong for already-customised
spaces. Cause: `OnGetAsync` built the section list from `GetBoard(id)`, which
returns the **custom** board — so for any customised space `BoardSpace.Name` is
the *custom* name, and that was being used as `SpaceCard.DefaultName`.

Fix: build the sections from `GetBoard(null)` — the default board (`skinId =
null`). Default and custom boards share identical indexes, colours and space
types (only names differ), so the layout is unchanged but `DefaultName` is now
correct. Customisations still come independently from `Skin.Spaces`, so
`DisplayName` (`Custom?.Name ?? DefaultName`) and `IsCustomised` keep working.

(For the record: this `GetBoard(id)` call predated this session's work — the
create/edit-flow change in §2 left that line untouched.)

## 8. Board sharing — the Share page

**Files:** `Pages/Boards/Share.cshtml`, `Pages/Boards/Share.cshtml.cs`,
`wwwroot/scripts/boards-share.js` (new), `Pages/Boards/Index.cshtml`

A dedicated **`Boards/Share/{id}`** page (route `@page "{id?}"`).

- Two-column layout in one form: **"Not shared with"** / **"Shared with"**, each
  a card with a live count badge.
- Each friend is a card: profile circle (`<profile-circle>` tag helper) +
  display name + green **Share** / red **Revoke** buttons (only the
  column-appropriate one shown).
- Each card carries a hidden `name="friendIds"` input. It is `disabled` while
  the card is in the "Not shared with" column and enabled in "Shared with" —
  disabled inputs are excluded from the POST, so the form submits exactly the
  shared set.
- `boards-share.js` — clicking anywhere on a card (or Enter/Space) moves it to
  the other column, flips the hidden input's `disabled`, swaps the Share/Revoke
  button, toggles a green border, and refreshes counts/empty placeholders.
- **Save** buttons top and bottom.

**Decision:** the disabled-hidden-input technique was chosen over collecting ids
in JS — the shared set stays plain form state, so a normal POST works with no JS
serialisation and antiforgery is handled automatically.

**Wiring to the service** (after John's service was reviewed and fixed — see
§9):
- `OnPostAsync` now calls `BoardSkinShareService.TryShareBoardSkin(id,
  friendIds)` and reports success/failure via the status message.
- **Preload:** `OnGetAsync` loads current recipients via
  `GetUserIdsForSharedBoardSkin(id)` into `SharedFriendIds`; the page renders
  already-shared friends straight into the "Shared with" column. This closes a
  **silent-revoke** risk — without preload, re-opening the page and saving would
  have treated all existing recipients as "removed" and revoked them.

**Index links:** a **Share** link was added on each owned board (card + list
views) pointing to `./Share`, so the page is reachable.

## 10. BoardSkinShareService — review cycles

John wrote `Services/BoardSkins/BoardSkinShareService.cs`; I reviewed it across
several rounds. Issues found (all fixed by John):

- **DI registration missing** — `BoardSkinService` took `BoardSkinShareService`
  in its constructor but the service was never registered → runtime resolution
  failure. Added `services.TryAddScoped<BoardSkinShareService>()`.
- **Soft-delete ignored** — `GetSharedBoardSkinsQuery` matched any `SharedWith`
  row, so a *revoked* (soft-deleted) share still showed the board. Added
  `!s.IsDeleted`.
- **No ownership check** — `ValidBoardSkin` only checked the skin existed, so
  any user could share/revoke *someone else's* board. Added `b.UserId ==
  _userInfo.UserId` (later made conditional via a `checkUserId` flag, since
  `TryRemoveSharedBoardSkin` needs to act on boards owned by *other* users).
- **`ValidUserIds` inverted (twice)** — the enabled-user check first read "are
  *all* enabled users in `userIds`?"; corrected to a count-compare. The friend
  check was then *also* inverted (`notFriends` actually meant "any are
  friends", and `return !notFriends` only passed when *none* were friends);
  corrected to a count-compare, and `f.DateRemovedUtc == null` added so
  un-friended users don't pass.
- **`GetSharedBoardSkinsQuery` over-constrained** — John briefly added
  `b.UserId == _userInfo.UserId`, which made "shared with me" require the board
  to be *owned by* me → always empty. Flagged; John removed it.

`GetBoardDropdown` TODO (in `BoardSkinService`): I pointed at the existing
`DropdownHelper.FromCollection` helper (used in `Friends/Index.cshtml.cs`); John
wired it up and the method now returns `Task<List<SelectListItem>>`.

## 15. Custom ruleset — domain knowledge

While reviewing `GamePlayer` I queried `TripleBonus` (a "triple" looked
impossible with two dice). John clarified the custom ruleset: `Dice1`/`Dice2`
hold the player's **assigned personal number** (rolled once at game start), not
the live roll; a turn uses **three dice** (two main + a third that moves every
other player, clockwise); the bonus fires when a player's number is re-rolled.
Saved to memory so this isn't re-flagged in future sessions.

**File:** `design-docs/game-rules.md`

Began writing up the custom ruleset, dictated section by section by John and

Clarifications resolved during dictation: doubles and triples symmetrically
reset each other's in-a-row counts; a triple also grants an extra roll;
Percentage-card tiers (100% double hotel / 50% any buildings / 10% none);

double roll; Free Parking "eligible to hand in" = a property not in a built-on
complete set whose set hasn't been handed in before.

Out of scope for this doc: per-deck **card contents** (what each individual
card does) — that's content, not rules, and will live elsewhere.
