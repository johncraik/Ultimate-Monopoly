## 1. 🌟 User Management — tiered auth hardening + peer-moderation

- **Auth matrix reminder** (design §4): **Admin + SystemAdmin** = change display name, apply/lift
  **Restricted**, **Disable/Enable**, hide/show (content). **SystemAdmin only** = grant/remove any role,
  **Delete**.
- **John pushed the tier checks down into `UserManagementService`** (not just the page): `SetRole` /
  `DeleteUser` early-return for non-SystemAdmin; `ApplyRole` blocks an Admin granting Admin/SystemAdmin.
- **Peer-moderation guard (John added):** a plain Admin **cannot Restrict or Disable a SystemAdmin
  target** (`SetRestricted` / `SetEnabled` re-check the target's role). I **mirrored it in the Details UI**
  with a `CanModerateTarget => IsSystemAdmin || !TargetIsSystemAdmin` gate — the Restrict / Enable-Disable
  buttons disable with a tooltip + note when a plain Admin views a SystemAdmin. Role-management + Delete
  were already wrapped in `@if (Model.IsSystemAdmin)`; the list table exposes no such actions.

## 2. 🌟 Role-change propagation — `AuthRefreshService` + middleware (design §6.2)

- **`AuthRefreshService`** (singleton, John): a `ConcurrentDictionary<string, byte>` of user ids —
  `RefreshUserSignIn` / `IsRefreshPending` / `ConsumeRefreshSignIn`.
- **`AuthRefreshMiddleware`** (I built): reads the **current** request's `IUserInfo.UserId`; if flagged,
  resolves the current principal's `AppUser` (`GetUserAsync(context.User)`) and calls
  `SignInManager.RefreshSignInAsync(user)`, then consumes the flag. Only ever refreshes the **requester's
  own** session — never a target's. Registered singleton in `AddAdminServices`; wired in `Program.cs`
  **immediately after `app.UseIdentity()`** (so `IUserInfo` is populated + principal authenticated).
- John wired `RefreshUserSignIn(targetId)` into the mutating `UserManagementService` methods.

**User Management is functionally done.** The only remaining Phase-2 item — `OrphanCleanupJob` — is
**deferred** (see §10).

## 3. 🌟 Reports — Phase 3 (Index + Details)

- **`ReportResolution` enum evolved** (John): `Open=0, Handled=1, AccountRestricted=2, AccountDisabled=4,
  AccountDeleted=8, FullAction(6), AllActions(14), AnyAction=-1`. Power-of-two so the list orders sensibly
  (`OrderBy(Resolution)`). **`Handled` is a genuine standalone resolution** (no-action close); the
  `Try*` methods **replace** the resolution from `Handled` (drop the flag) rather than OR-ing — so
  `Handled | Deleted` (the value that would clash with the old `AnyAction=9`) is never produced.
  `AnyAction=-1` is a never-stored filter sentinel.
- **Index** (mirrors users): single **Resolution** radio (Any · Open · Handled→relabelled **Resolved** ·
  Restricted · Disabled · Comprehensive · Deleted · All) + search. **Filter-semantics fix:** the service
  default branch was `!= Open` for the single flags → changed to exact `== resolution`; `Handled` = all
  non-open; `Any` = everything.
- **Table:** ReportId · Reported User ID · Reported User (name + view-user eye, right-aligned) · Reason ·
  Message (trunc 50) · Resolution · **Reported By User ID** · **Reported** (date) · Open. John added the
  **raw `ReportedUserId` / `ReportedByUserId` + `ReportedByUser` + `ReportedDate`** to the VM; id columns
  use raw ids (show for deleted users too), fixed-width (`admin-id-col`). **Open** column = amber
  exclamation-circle (open) / grey dash (resolved). Sidebar: **Open / Resolved / All** links.
- **Details (WIP):** two-column — left = report info; right = **Reported By** short card **above** the

## 9. CSS iteration log (Game Details panel)

`.admin-game-state` went through: `90vh` + internal scroll → tab-bar `fixed` (lives in the live drawer via
a transformed `.player-drawer`; pinned to the viewport here) → `absolute` (didn't behave) → **`sticky`**
(worked, but floated up on short content) → flex-fill + `margin-top:auto` (pinned on short content) →
**final**: with content now stacked below the profile, dropped the flex-fill; `sticky bottom:0` (no
margin) + sticky header `top:0`. The transform-containing-block insight (a `position:fixed` child pins to
the nearest **transformed** ancestor, else the viewport) is the key fact behind the live drawer's behaviour.

## 10. Decisions locked this session

- **Peer moderation:** a plain Admin can't Restrict/Disable a **SystemAdmin** target (service + UI). Other
  Admins remain moderatable.
- **Role-change propagation:** singleton `AuthRefreshService` + `AuthRefreshMiddleware` after `UseIdentity`;
  only ever refreshes the requester's own session.
- **Background jobs → a dedicated final v1-roadmap phase** — `OrphanCleanupJob`, `GameRetentionJob`, log
  retention are scoped/configured together then; build the synchronous half now, **don't re-flag the missing
  job**. (Memory saved.)
- **`ReportResolution`:** `Handled=1` is a real resolution (replaced, not OR'd, on action); `AnyAction=-1`
  is a non-stored sentinel; filter = exact match (single flags), `!= Open` (Handled), all (Any).
- **Rules editor** edits title/description/hidden only; `IsHidden` not yet consumed player-side.
- **Turn tax:** % rates, brackets fully-set-or-zero, ascending thresholds, Disable = zero all six.
- **Game actions** are NOT wired — they need **admin-callable `GameService` paths** (the host actions are
  player/host-scoped). The intended call **+** the `AdminActionLog` write live in `GameManagementService`.
- **Game-state read-only flag = `IsAdminView` on the model** — **not** ViewData (which doesn't reach a
  handler-returned partial), **not** a viewer-is-player inference.
- **admin → game asset reuse is fine; game → admin is never.** Reused partials need **absolute** nested-
  partial paths. Extracted `_GameInfoPanel` rather than duplicate it.
- **Downloads** = single JSON document, snapshot + events embedded as nested JSON, for inspection.
