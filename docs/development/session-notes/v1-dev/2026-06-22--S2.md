## 1. 🌟 Engine — "advancing to a space doesn't DRAW a card" (reviewed John's impl)

The missing rule: a player **advanced** (by a card) onto a card-drawing corner/tax space does **not**
draw that space's card — **except Chance / Community Chest**, which still draw. Normal dice landings
are unaffected.

- **Mechanism:** a per-turn "prevent" registry on `GameCacheModel` (`Prevent` / `ClearPrevent` /
  `PreventBoardIndexCard(playerId, index)` — check-and-consume). `[JsonIgnore]`, cleared at the turn
  boundary inside `UpdateMetadata` (John moved the `ClearRuleCodes` + `SetTurnState(StartOfTurn)` there
  too, so all three transitions share it). Cache-only is correct per the restart contract — register
  and consume both happen within one turn's execution.
- **`MovementService.AdvancePlayer`** registers the prevent for GO / JustVisiting / FreeParking /
  GoToJail / Tax destinations — **only** the advance path (`MovePlayer`, incl. third-die, never does).
  Chance/ComChest deliberately excluded → they still draw.
- Each space service consumes the flag before drawing: `BoardService` (JustVisiting), `GoService`,
  `FreeParkingService`, `TaxService`, `JailService.GoToJail` (wraps the whole immunity+draw block).
- **John's `willResolveSpace` fix** (after my review): `AdvancePlayer` takes `willResolveSpace = true`,
  threaded from `MovementActionService`'s `action.ResolveLandedSpace`, so the prevent is registered
  **iff the destination will actually be resolved** — closing the stale-entry footgun. The two
  non-resolving advance paths (leave-jail → JustVisiting; the `GoToJustVisiting` `continue` branch)
  now pass `willResolveSpace: false`.
- **Key facts confirmed:** `JailSpace = 100` (JustVisiting × 10) ≠ `JustVisitingSpace = 10`, so
  `SendPlayerToJail` never registers a spurious prevent. The swap example John worried about never hit
  the prevent path anyway (`ApplySwap` sets `BoardIndex` directly, no `AdvancePlayer`).

Engine + web build clean throughout.

## 3. 🌟 A2 — Roles + `Restricted` (finished, 🟢)

`AppRoles.Restricted` (+ `RestrictedDesc`) seeded via `ConfigureAdminAndRolesAsync<AppUser, AppRole,
AppDbContext, AppRoles>` (the `{Name}`/`{Name}Desc` reflection convention). **Boundary decided:** a
restricted account **cannot** send friend requests, send messages, create games, or create/share board
skins — everything else (accept/decline/remove friends, join games, edit existing skins, remove existing
shares) stays open.

**The one hole I found:** create-board-skin was **only UI-gated** (`Boards/Edit.OnGet → Unauthorized`) —
`BoardSkinService.TrySaveSkin` had no server guard, so a crafted POST could create a skin. **John added**
the authoritative guard in the `TrySaveSkin` create branch. Every other gated action already guards
server-side (`TryCreateNewGame`, `TrySendFriendRequest`, `TryShareBoardSkin` — which correctly blocks
*add/restore* but allows *remove*). The **messages** restriction is declared but enforced at E1 (no
messaging surface yet). Marked **A2 🟢** on the roadmap.

## 4. 🌟 E2 — Hidden profile / soft-block (→ ◐ partial)

John implemented it as a **role (`HiddenUser`), not a `HiddenProfile` flag** — reuses the seeded-role
infra (no migration), gives C1 a toggle for free. Then **improved it to hide-from-non-friends-only**:
`ProfileService.GetHiddenUserIds` (excludes self) + `LeaderboardService` anonymises (the "Unknown" VM,
W/L/D kept so still ranked) **only when `!friend && hidden`** — self and friends see normally. Enforced
by a fresh `UserRoles` DB query, **not claims**, so a toggle takes effect immediately (no cookie
refresh). Set methods: `TryHideUser` / `TryUnhideUser` (self-service, **SystemAdmin** to act on others,
idempotent).

**Critical bug I caught:** the improved `GetHiddenUserIds` hardcoded `r.Name == "Hidden"` (≠ the seeded
`"HiddenUser"`) with `SingleAsync` → threw on every leaderboard load. **John fixed** to
`FirstOrDefaultAsync(r => r.Name == AppRoles.HiddenUser)` + a null guard. Marked **E2 ◐ PARTIAL** — the
toggle now lives on the Account tab; **remaining: add the same toggle to `/Profile/Index`**.

## 5. 🌟 B1 — Profanity filter + display names (built end-to-end, 🟢)

Designed with John, then built. **Decisions locked:** comprehensive **`Profanity.Detector`** library
(NuGet id has a dot — plain `ProfanityDetector` doesn't exist; 0.1.8, ~1.7M downloads) **+** a private
DB-backed local list; **no allow-list table for V1** (lean on the library's own); **under-block + admin
override**; **generic** user message (term → audit only, not echoed); display-name **collisions allowed**.

Pieces (placed per John's structure after I first mis-placed them):
- **`Helpers/ProfanityNormaliser`** — *static* helper. Canonical form: strip diacritics, leet-map,
  drop separators/whitespace, collapse 3+ repeats. Same transform feeds `BlockedWord.NormalisedWord`.
- **`Records.cs` `ProfanityResult`** record; **`Enums/ProfanitySource`**.
- **`BlockedWord`** entity (`Word` / `NormalisedWord` PK, `AuditModel`) — John created entity + migration.
- **`Services/Imports/BlockedWordImportService`** — additive seed from `BlockedWords_FilePath` (one
  word/line, `#` comments, never deletes; audits as `IUserInfo.SYSTEM_ID`), invoked from
  `SetupDefaults`.
- **`Services/Cache/BlockedWordsCacheService`** — `IMemoryCache` NeverRemove, hydrate-on-miss,
  SystemAdmin-gated `Invalidate`.
- **`Services/ProfanityService`** (scoped, root) — `Check` → library (raw) ∪ library (normalised) ∪
  local-list substring.
- DI in `ServiceRegistration` (library filter singleton; cache/orchestrator/import scoped).
- **Wired into write paths:** register (username) + display-name set/change (Account tab). Generic
  "this name isn't allowed" message.

**Case-sensitivity bug (live):** "Shit FUCKER" was allowed — `Profanity.Detector` is **case-sensitive**
(lowercase list). John added `input.Trim().ToLower()` to the raw pass. **Still open:** the *normalised*
library pass passes the **uppercase** `Normalise(...)` output, so it's dead — needs `ToLowerInvariant()`
on that call to catch leet-evasions of common words (`sh1t`, `f.u.c.k`) via the library. (Use
`ToLowerInvariant` over `ToLower` generally — Turkish-İ.) Marked **B1 🟢**.
