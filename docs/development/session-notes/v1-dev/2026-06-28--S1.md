## 3. 🌟 Built E1 end to end

- **Config** — `AddMessaging<AppDbContext>(o => o.DisableGroups = true)`.
- **`MessagingHub`** (`/hubs/messaging`) — `[Authorize]`, **push-only**, no client→server methods. Crucial
  realisation: **no manual groups needed** — SignalR's `Clients.User(userId)` routes by `Context.UserIdentifier`
  (the same mechanism `PresenceHub` already relies on). All pushes originate server-side from the service via
  `IHubContext<MessagingHub>`.
- **`FriendMessagingService`** (`Services/Friends/`) — the **guard seam**. Every method validates
  **friends-only** (`FriendService.AreFriends`) · **not blocked** both ways (`BlockAndReportService.CheckIfBlocksExist`)
  · **`Restricted` → read-only** (`_userInfo.IsInRole(AppRoles.Restricted)`) before delegating. `GetConversations`
  resolves the other participant → `UserProfileViewModel`, computes unread from `MessageReadLog`; `OpenOrCreate` /
  `LoadThread` / `SendMessage` (pushes on send). `BuildView` also computes the **read receipt** (my latest message,
  read by them).
- **`/Social/Messages`** — two-pane page (`Index` + `_Conversation` partial + `messages.js`): AJAX thread-load
  (`?handler=Thread`), AJAX send (`?handler=Send`, antiforgery header), live `ReceiveMessage` append + list
  bump/unread, `<profile-circle>` avatars, restricted-user disabled footer. Colour-variable compliant.
- **Wiring** — "Message" button on `Friends/Profile`; friends-list `Message` buttons repointed to the new page
  (`FriendViewModel.FriendId`); navbar "Messages" link live; **mock `Friends/Chat` deleted** + its dead
  `ChatMessage` view-model record removed from `Records.cs` (it shadowed JC's `ChatMessage`).

## 4. 🌟 Polished from a live run (John's three fixes)

1. **Sidebar truncation** — the name/preview weren't ellipsis-truncating. Cause: `min-w-0` **isn't a stock
   Bootstrap 5 class**, so the flex child never shrank. Fixed with real CSS (`min-width: 0` on `.msg-list-item`
   + its `.flex-grow-1`).
2. **Fixed-height scroll** — the panel grew with messages instead of scrolling. Root cause: the `card h-100`
   percentage had **no definite parent height to resolve against**. Fixed by cascading the height
   **row → column → card** (`.msg-shell > [class*="col-"] { height: 100% }` + `.msg-shell .card { height: 100% }`),
   then `.msg-body { flex: 1 1 auto; min-height: 0; overflow-y: auto }`. Wrapped the fixed height in
   `@@media (min-width: 768px)` so mobile (stacked panes) flows naturally. (`@@media` — Razor escaping in a
   `<style>` block.)
3. **Read receipt** — small grey "**Read HH:mm**" under my latest message when the friend has read it (via
   `MessageReadLog`); `messages.js` strips the stale receipt when a newer message is appended.

## 5. Decisions locked this session

- E1 is **DM-only, friends-only, no metadata**; guards (friends/block/`Restricted`-read-only) live in
  **our** `FriendMessagingService`, never in the package or the hub.
- **SignalR push via `Clients.User`** (no per-user groups) — the reusable pattern for any future per-user hub.
- New chats start from a **profile / friends-list** "Message" button; the Messages page lists existing
  conversations only (no in-page picker). Mock `Friends/Chat` **retired**.
- Messaging's package **cleanup jobs are deferred** to the all-package-jobs pass.
