## 2. Filling out the prompt catalogue

Worked through `game-rules.md` to identify every mid-execution decision
point that still needed a prompt. Final list ‚Äî three landed in the first
pass, two more later:

### 2.1 Generic selectors (foundation primitives)

- **`TargetPlayerPrompt`** ‚Äî pick `Count` players from `EligiblePlayerIds`.
- **`TargetPropertyPrompt`** ‚Äî pick `Count` properties from
  `EligibleBoardIndexes`.

Key decision (John's): drop `MinCount`/`MaxCount` in favour of a fixed
`Count`. There is never a choice of *how many* to pick ‚Äî the calling
context always knows. Where cards in the same deck differ in target count
(some target 1 player, others target 2), they remain separate cards each
with their own fixed `Count`.

### 2.2 `ShortfallPrompt`

The choice when a player can't pay. Response is a single `ShortfallAction`
enum: `TakeLoan` / `Mortgage` / `SellHouses` / `ProposeDeal` /
`DeclareBankruptcy`.

Key rule clarifications that came out of designing this:

1. **Sell-houses is both a portfolio command *and* a shortfall response.**
   Default rule 7 only restricts using either route to *raise funds for a
   new commitment* (buy / bid). At idle you can sell freely.
2. **Voluntary bankruptcy IS a turn-idle action**, not literally
   "at any time" ‚Äî `game-rules.md` Bankruptcy rule 1's "at any time"
   means "at any turn boundary", not "mid-roll".
3. **Deals during a shortfall must be with the creditor.** A deal with
   the player you owe settles the debt directly (the deal IS the
   payment). Deals with non-creditors to raise funds are blocked under
   the same fairness principle as rule 7.

Drove a `game-rules.md` update: Default rule 7 extended from "no
mortgaging to buy/bid" to "no raising funds via mortgaging, selling
buildings, or non-creditor deals" with the creditor-deal exception
explicitly called out.

### 2.3 `ShortfallPrompt` fields ‚Äî Cost + PlayerBalance + computed AmountOwed

Started with just `AmountOwed`. John refactored to carry `Cost` (total
owed) + `PlayerBalance` (cash available); `AmountOwed` became
`Cost - PlayerBalance` as a get-only computed property.

First version had `[JsonIgnore]` on it. Discussion: should it serialise?
Conclusion: YES ‚Äî get-only properties serialise out (frontend gets the
pre-computed shortfall) and are skipped on deserialise (no setter ‚Üí
nothing to poison). The get-only nature is the safeguard. `[JsonIgnore]`
dropped.

### 2.4 `AuctionBidPrompt` and `CardOptionPrompt`

- **`AuctionBidPrompt`** ‚Äî sequential bid loop per `game-rules.md`
  Default rule 6. Fields: `BoardIndex`, `CurrentHighBid`,
  `CurrentHighBidderId?`, `PlayerBalance`. Response is `AuctionBidAction`
  (Bid/Pass) + nullable `BidAmount`. Validator enforces bid >
  CurrentHighBid AND ‚â§ PlayerBalance (Default rule 7 ‚Äî genuine cash only).
- **`CardOptionPrompt`** ‚Äî generic n-ary labelled selector. Options are
  `(Key, Label)` records; response carries `SelectedKey`. Stable keys
  rather than indexes so logs stay meaningful and card reorders don't
  silently change historical meaning.

Catalogue (¬ß15 of `choice-events.md`) closed out ‚Äî no more prompts
currently planned.

## 3. `PlayerId` promoted to `Prompt` base

Observation: 6 of 7 implemented prompts had a `PlayerId` field plus an
identical `Target => SinglePlayer(PlayerId)` override.

First attempt overreached ‚Äî moved both `PlayerId` *and* defaulted
`Target` on the base. John pushed back: `PlayerId` and `Target` are
*different concepts* that happen to coincide for single-player prompts.

Final answer:

- **`PlayerId`** = the *subject* of the prompt (the player it's about).
  Promoted to base, producer-set via `init`.
- **`Target`** = the *audience* (who sees it / who can respond).
  Stayed abstract ‚Äî each concrete prompt declares its own.

For `InterruptibleWindowPrompt` (group-targeted), the inherited
`PlayerId` carries the *initiator* of the play the window is interrupting
‚Äî useful for UI rendering ("Player X played card Y ‚Äî anyone NOPE?").

Concrete realisation that the subject/audience distinction is real even
though it coincides for single-player prompts: a future prompt could
legitimately be about one player but rendered/responded by the host
alone. Forcing `Target` to default off `PlayerId` would close that door.

## 5. `TurnStateProvider` ‚Äî design and skeleton

Designed and built the provider (foundation, no game logic). Two parts:

### 5.1 Capability gates

Composed from primitives (`IsEngineIdle` / `IsCurrentPlayer` / `IsJailed`
/ `IsAtTurnBoundary`):

| Command group | Gate |
|---|---|
| Portfolio (mortgage / build / sell / play card / pay loan early) | StartOfTurn only, current player, not in jail, engine idle |
| Deal (propose / accept) | Either turn boundary, engine idle |
| Jail exit | StartOfTurn, current player, in jail, engine idle |
| End turn | EndOfTurn, current player, engine idle |
| Voluntary bankruptcy | Either turn boundary, engine idle |

Key John call: **portfolio is `StartOfTurn` only, not both boundaries.**
"You can't build/sell/mortgage/play a card at the end of your turn." So
EndOfTurn is a narrow window for only deals, bankruptcy, and the end-turn
command itself.

### 5.2 Named transitions

Replaced the broken linear `ProgressTurnState` with explicit branched
transitions. The breakthrough piece ‚Äî and the most important design
decision in the session ‚Äî was the realisation about extra turns:

**Extra turn fires from EndOfTurn, not from the movement phases.**

The chain: PlayerRollMovement ‚Üí ThirdDieMovement ‚Üí EndOfTurn ‚Üí *either*
ExtraTurn OR NextPlayer. Routing extra turns through EndOfTurn means
deals (and voluntary bankruptcy) can happen between every roll, not just
between players.

John's framing: an extra turn IS a new turn (full portfolio window,
fresh deal opportunity, new event window, new snapshot row) ‚Äî but the
*same player* and the *same `TurnNumber`*.

That distinction drove the GameTurn vs GameSnapshot clarification:

- **`GameTurn`** ‚Äî the "whose turn is it" record (`CurrentTurnId`,
  `TurnNumber`, `CurrentPlayerId`). Created exactly once per
  player-advance.
- **`GameSnapshot`** ‚Äî the per-state snapshot. **1:many** with
  `GameTurn`. Multiple snapshot rows sit under one GameTurn when extra
  turns fire.

This contradicts `game-engine.md` ¬ß8 ("1:1 with shared primary key") ‚Äî
flagged in `turn-state.md` ¬ß9 as a doc fix needed.

Final transitions:

- `TransitionToRollPhase()` ‚Äî StartOfTurn ‚Üí PlayerRollMovement.
- `TransitionToThirdDie()` ‚Äî PlayerRollMovement ‚Üí ThirdDieMovement.
- `TransitionToEndOfTurn()` ‚Äî from PlayerRoll (triple skip / 3-in-a-row)
  or ThirdDie (normal) ‚Üí EndOfTurn.
- `TransitionToExtraTurn(bool isTriple)` ‚Äî EndOfTurn ‚Üí StartOfTurn
  (same player). Bumps the matching `DoublesInRow` / `TriplesInRow`
  counter, resets the other (Doubles/Triples rule 6). Returns `GameModel`
  for snapshot persistence.
- `TransitionToNextPlayer()` ‚Äî EndOfTurn ‚Üí StartOfTurn (next player).
  AdvancePlayer logic stubbed (skip bankrupt, decrement TurnsToMiss, new
  CurrentTurnId, bump TurnNumber) ‚Äî borders on game logic, deliberately
  TODO. Returns `GameModel`.

### 5.3 The transitions-as-prompt-handlers detour

John floated: "should `TransitionToRollPhase` send the dice prompt,
check for double/triple, send to jail on 3-in-a-row, and return the
dice?". Pushed back ‚Äî that's turn-loop orchestration, not a transition.
The provider stays foundation; prompts + rule decisions + jail-sending
live higher up. John agreed.

Useful principle established: **mutations intrinsic to the state change
belong in the transition method.** A snapshot-producing transition owns
whatever mutations are needed to make the snapshot correct. Counter
bumps in `TransitionToExtraTurn` fit. Sending the player to jail does
not ‚Äî that's a conditional consequence of one specific roll outcome,
not intrinsic to "the roll phase".

## 6. Event receipts framework

Pivoted to the event-receipts framework ‚Äî the engine's output channel,
parallel to prompts being its input channel.

### 6.1 What they ARE ‚Äî the framing John pinned

Receipts are "transactions of what happened" ‚Äî state-change records
emitted after the fact. Importantly, they're the source data for the
(future) statistical-snapshot projection: stats like "most money earnt",
"highest earning property", "times in jail", "doubles/triples count"
all compose from the per-turn receipt stream.

### 6.2 The granularity question

The existing code had a mix of primitive receipts
(`FinancialTransactionReceipt`, `PropertyTransactionReceipt`) and
semantic ones (`PlayerLeftJailReceipt`, `FreeParkingReceipt`,
`PlayerBankruptedReceipt`). Inconsistent.

Resolution: **semantic-flavoured receipts, primitive-rich content.**
Each receipt's *type* answers "what kind of thing happened"; the *fields*
carry enough primitive data that stats can run direct sum/count
aggregations without re-implementing rule logic.

### 6.3 Concrete changes

- **Drop `PlayerLeftJailReceipt`** ‚Äî decomposes to `FinancialTransaction
  (Reason=JailFee)` + `PlayerMoved`.
- **Drop `FreeParkingReceipt`** ‚Äî decomposes to
  `FinancialTransaction(Reason=FreeParkingPay/Take)`.
- **Drop `EventReceiptType` enum** ‚Äî dead duplicate of the
  `JsonDerivedType` discriminator list.
- **Add `PlayerEnteredJailReceipt`** ‚Äî semantic receipt for the
  jail-count stat. Inferring it from `PlayerMoved.FinalBoardIndex ==
  JailSpace` would be stats encoding rule logic.
- **Enrich `FinancialTransactionReceipt`** with `Reason` (18-value
  `FinancialReason` enum: Rent / Tax / Fine / GoBonus / JailFee /
  FreeParkingPay/Take / LoanTake/Repay / Purchase / Auction / Build /
  Sell / Mortgage / MortgageFee / Unmortgage / CardPayout / CardCharge /
  Deal) and `SourcePropertyId?` for property attribution.
- **Add base fields** `TurnNumber` and `SequenceIndex` (both
  `{ get; internal set; }` ‚Äî assigned by `cache.AddEvent`, not by the
  producer; `init` wouldn't allow framework backfill).
- **Switch all receipts to `init`** for immutability of producer-set
  fields.
- **`IEventEmitter` seam** ‚Äî parallel to `IPromptProvider`. Producer
  emits via `IEventEmitter.Emit(receipt)`; concrete `EventEmitter`
  forwards to `cache.AddEvent`. Registered scoped in
  `ServiceRegistration`.

## 7. Tests ‚Äî three suites, 103 tests

Pivoted to tests for the foundation framework. Three test classes in
`MP.GameEngine.Tests/FrameworkTests/`:

### 7.1 `PromptProvider_Tests` (27)

Grouped by region: RequestAsync setup, RequestAsync async, TrySubmit
failure paths (√ó 7 ‚Äî stamp / pending / id / type / auth + side-effect
absence), TrySubmit success path (√ó 5), interaction, cancellation,
concurrent submissions.

**Surfaced two real bugs:**

1. **Cancellation didn't clear the cache's pending prompt.** Cancelling
   the token cancelled the TCS but left the prompt stranded in the cache.
   A subsequent TrySubmit would "succeed" silently against a dead TCS.
   Fixed by wrapping the await in `try/finally`; finally clears pending
   if it's still ours (id check guards against future races). Re-stamping
   in `ClearPendingPrompt` invalidates any concurrent submission.
2. **Concurrent submissions both returned true.** Two parallel valid
   submissions could pass all checks, both call ClearPendingPrompt +
   TrySetResult, and both return true (second's TrySetResult silently
   no-ops). Fixed by reordering: TrySetResult is now the atomic gate ‚Äî
   only one of any concurrent submitters can transition the TCS, losers
   report false.

Both tests un-skipped after the fix; suite at 27 green.

### 7.2 `TurnStateProvider_Tests` (65)

Grouped by method. Theory rows expand the suite where wrong-state
exhaustively. Coverage: all five capability gates (happy + every
failure axis), all five transitions (success state, restamp, throws on
wrong state), extra-turn counter behaviour (four tests for the four
combinations of bump-this/reset-that), events clearing on extra turn
and next player.

`AdvancePlayer` assertions deliberately omitted ‚Äî it's a stub. Tests
to be added when implemented.

### 7.3 `EventEmitter_Tests` (11)

Grouped by concern: forwarding, TurnNumber bookkeeping, SequenceIndex
bookkeeping (first is 0, increments, resets on ClearEvents), ordering
and mixed types, concurrency stamping.

## 9. Architecture observations worth carrying forward

- **The framework set is now functionally complete for the foundation
  layer.** Prompts (input), turn-state (gating + transitions), event
  receipts (output). Each has a design doc, code, and tests. Nothing
  consumes them yet ‚Äî that's the next layer up (rule services + turn
  loop).
- **The pattern across all three frameworks is consistent.** Each has
  an `I‚Ä¶Provider` / `I‚Ä¶Emitter` seam, a concrete implementation against
  `GameCacheModel`, a `Models/<name>/` directory with the typed payloads,
  and a single registration line in `ServiceRegistration.AddGameEngine`.
  Rule services will plug into all three the same way.
- **The cache is the integration point.** Pending prompt, event list,
  game model, turn state, dice roll ‚Äî all live on `GameCacheModel`. The
  providers / emitter own the mutations; the cache enforces the stamp.
- **Mutations intrinsic to a transition belong on the transition.** From
  the `TransitionToExtraTurn` discussion ‚Äî the principle that decides
  what does and doesn't belong on a foundation primitive.
- **Subject vs audience is a generally useful distinction.** Came up
  for prompts (`PlayerId` vs `Target`) and applies again for receipts
  (`PlayerId` is the subject; counterparties live on typed fields).
